# 比較評価ルーブリック

## 1. 評価方針

この比較は、最終的なゲームの派手さを競うものではない。

主目的は、AIが同じ仕様を実装した時に、どのライブラリ/エンジンが作りやすいかを観察することである。

実装方式は、単一エージェントによる連続実装とする。

このため、評価では「そのライブラリ単体の作りやすさ」と「前回までの実装から得た知見による作りやすさ」を分けて扱う。

評価では以下を分けて考える。

- 実装前のセットアップ容易性
- 実装中のAPI理解しやすさ
- ゲーム構造の自然さ
- 衝突判定/物理の扱いやすさ
- UIの扱いやすさ
- デバッグしやすさ
- 完成物の安定性
- コードの読みやすさ
- 前実装からの流用しやすさ
- 前実装の設計が邪魔にならなかったか

## 2. 評価スケール

各項目は `1` から `5` で評価する。

| 点 | 意味 |
| ---: | --- |
| `1` | かなり難しい。AIが迷いやすく、修正コストも高い |
| `2` | やや難しい。成立はするが、落とし穴が多い |
| `3` | 普通。明確な長所短所がある |
| `4` | 作りやすい。多少の注意で自然に実装できる |
| `5` | 非常に作りやすい。仕様から実装までの距離が短い |

## 3. 評価項目

### 3.1 セットアップ

見る点:

- 依存関係の導入が簡単か
- Vite等との相性が良いか
- TypeScript利用時に型が安定しているか
- 最初の画面表示までが短いか
- サンプルやドキュメントの導線が分かりやすいか

記録例:

```text
Score: 4
Reason: npm + Viteで即起動できた。初期化コードも短い。
```

### 3.2 ゲームループ/Scene管理

見る点:

- update loopが分かりやすいか
- deltaTimeの扱いが自然か
- ゲームオーバーやリスタートの状態管理がしやすいか
- エンティティの追加/削除が安全か

### 3.3 入力

見る点:

- キーボードの押下状態を扱いやすいか
- マウス座標をワールド座標へ変換しやすいか
- Canvas/WebGLの座標ずれに悩まないか
- 射撃の押しっぱなし処理が簡単か

### 3.4 描画

見る点:

- 図形だけでゲーム画面を作りやすいか
- 座標系が分かりやすいか
- HUDとゲームオブジェクトを自然に重ねられるか
- 2D/3D表現が仕様に合うか

### 3.5 衝突判定/物理

見る点:

- 円、矩形、境界の衝突を実装しやすいか
- 物理エンジンが過剰または不足にならないか
- 衝突イベントの発火条件が分かりやすいか
- 手動判定との使い分けが自然か
- 障害物で詰まった時の挙動を制御しやすいか

### 3.6 敵AI/スポーン

見る点:

- 敵の生成と破棄が簡単か
- 追跡ロジックを素直に書けるか
- オブジェクト数が増えても管理しやすいか
- 難易度曲線を入れやすいか

### 3.7 UI/HUD

見る点:

- テキスト表示がしやすいか
- HPバーなどの簡単なUIを作りやすいか
- DOMとCanvas/WebGLの併用が必要か
- ゲームオーバー表示を重ねやすいか

### 3.8 リスタート/破棄

見る点:

- ゲーム状態を初期化しやすいか
- エンティティをまとめて破棄しやすいか
- イベントリスナーや物理Bodyが残らないか
- 再スタート後に重複更新が起きないか

### 3.9 デバッグ容易性

見る点:

- エラーが読みやすいか
- オブジェクト状態を確認しやすいか
- ライブラリ内部エラーと実装ミスを切り分けやすいか
- ブラウザDevToolsで追いやすいか

### 3.10 AI実装適性

見る点:

- API名が直感的か
- 古い情報と新しい情報の差で迷いやすくないか
- 最短実装と推奨実装の距離が近いか
- 小さな修正で壊れにくいか
- 自然言語の仕様からコードへ落とし込みやすいか

### 3.11 連続実装での差分

見る点:

- 前回までの仕様理解がそのまま活きたか
- 前回までのコード構造を持ち込んでも自然だったか
- ライブラリ固有の流儀に合わせ直す必要があったか
- 学習効果で実装時間が短くなったか
- 学習効果では埋まらない固有の難しさがあったか
- 前回の設計判断が逆に邪魔になったか

## 4. 重み付け

総合評価を出す場合は、以下の重みを使う。

| 項目 | 重み |
| --- | ---: |
| セットアップ | `10` |
| ゲームループ/Scene管理 | `15` |
| 入力 | `10` |
| 描画 | `10` |
| 衝突判定/物理 | `15` |
| 敵AI/スポーン | `10` |
| UI/HUD | `10` |
| リスタート/破棄 | `10` |
| デバッグ容易性 | `5` |
| AI実装適性 | `15` |
| 連続実装での差分 | `10` |

合計は `120` である。

点数化する場合は以下。

```text
weightedScore = sum(score * weight)
maxScore = 5 * sum(weight)
percentage = weightedScore / maxScore * 100
```

ただし、点数は補助情報とする。最終判断では、用途別の向き不向きを重視する。

## 6. 実装時間の記録

可能であれば、各実装で以下を記録する。

| 記録 | 説明 |
| --- | --- |
| Start time | 実装開始時刻 |
| First render | 画面に何か表示できた時刻 |
| First playable | 移動と射撃ができた時刻 |
| Core complete | 必須仕様が揃った時刻 |
| Fix time | バグ修正に使った時間 |
| Total time | 合計時間 |

厳密でなくてよい。AI実装の体感比較に使う。

## 7. 連続実装ログ

各実装の終了直後に、以下を記録する。

```md
## Sequential Notes

### From Previous Implementation

### Reused Successfully

### Reused Poorly

### Unique To This Library

### Bias Notes
```

`Bias Notes` には、実装順序の影響が疑われる点を書く。

例:

- Phaserで作った定数設計をそのまま使えたため、実装時間が短くなった
- 手動衝突判定の発想を物理エンジン側へ持ち込みすぎた
- 3D座標変換で、2D実装の直感がそのまま使えなかった

## 8. 手動テスト手順

各実装で最低限以下を確認する。

1. ページを開く
2. プレイヤーが表示される
3. `WASD` または矢印キーで移動する
4. マウス方向に照準が向く
5. 左クリックまたは `Space` で弾が出る
6. 敵が外周から出る
7. 敵がプレイヤーに近づく
8. 弾が敵に当たると敵が消える
9. スコアが増える
10. 障害物にぶつかる
11. 敵に触れるとHPが減る
12. HPが0になるとゲームオーバーになる
13. `R` で再開できる
14. ブラウザコンソールに致命的エラーが出ていない

## 9. 評価コメントの型

最終的な比較コメントは以下の型にする。

```md
## <Project Name>

### Summary

### Best Fit

### Strengths

### Weaknesses

### AI Implementation Friction

### Carryover Effect

### Notable Bugs or Workarounds

### Verdict
```

## 10. 最終比較で出したい結論

最終レポートでは、単一の「最強」を決めるのではなく、以下のように用途別に整理する。

- 最も短時間で完成しやすい
- 2Dアクションに向く
- 物理込みの2Dに向く
- 3D化しやすい
- 学習用に向く
- 小規模ゲームジャムに向く
- 長期保守に向く
- AIに任せやすい
- AIが躓きやすい
- 連続実装で学習効果が大きい
- 初見実装でも迷いにくい

この分類により、ゲームの種類や開発体制に応じた選定ができるようにする。
