# 単一エージェント連続実装計画

## 1. 目的

1つのエージェントが `Arena Core` を複数ライブラリで順番に実装し、実装体験の差を同じ文脈で比較する。

この方式では、後半の実装ほど前半の知見を利用できる。したがって、比較では以下を分けて記録する。

- ライブラリ自体の作りやすさ
- 前回までの実装から流用できた部分
- 前回までの実装が逆に邪魔になった部分
- 連続実装だから見えた差分

## 2. 実装順序

推奨順序は以下とする。

| 順番 | フォルダ | 理由 |
| ---: | --- | --- |
| 1 | `phaser` | 2Dゲームフレームワークの基準線を作る |
| 2 | `excalibur` | Phaserと近い2Dエンジンを、TypeScript寄りの設計で比較する |
| 3 | `kaplay` | 軽量で抽象度の高いAPIがどこまで楽かを見る |
| 4 | `kontra` | 軽量Canvasライブラリで、手動実装量との差を見る |
| 5 | `melonjs` | HTML5ゲームエンジンとしての標準機能と癖を見る |
| 6 | `pixijs-matter` | 描画ライブラリと物理エンジンの同期コストを見る |
| 7 | `threejs-rapier` | 2D仕様を3Dライブラリ + 物理へ移す負担を見る |
| 8 | `babylonjs` | 3Dエンジンとしての統合度とゲーム実装の自然さを見る |
| 9 | `playcanvas` | Web向け3Dエンジンをコードファーストで扱う負担を見る |

この順序は、まず2Dの基準を作り、次に軽量系、描画 + 物理の組み合わせ、最後に3D系へ進む構成である。

## 3. 1プロジェクトあたりの進行

各プロジェクトは以下の順序で進める。

1. 該当ライブラリの現行セットアップを確認する
2. 最小構成で画面を表示する
3. プレイヤー移動を実装する
4. 照準と射撃を実装する
5. 敵スポーンと追跡を実装する
6. 衝突判定または物理を実装する
7. 障害物を実装する
8. HUDを実装する
9. ゲームオーバーとリスタートを実装する
10. 起動確認と手動テストを行う
11. `IMPLEMENTATION_NOTES.md` を書く
12. 次プロジェクトへ進む前に比較メモを残す

## 4. 実装前チェック

各プロジェクト開始時に以下を確認する。

- 対象フォルダが正しい
- 前プロジェクトの実装が完了している
- 前プロジェクトの `IMPLEMENTATION_NOTES.md` が埋まっている
- 共通仕様の数値を再確認した
- 今回のライブラリで使う予定の主要APIを把握した

## 5. 実装後チェック

各プロジェクト完了時に以下を確認する。

- 必須機能が動く
- 1分程度の手動プレイで致命的エラーがない
- READMEに起動方法がある
- 実装メモに差分がある
- 実装メモに学習効果の記録がある
- 次の実装へ持ち込む設計判断が明確である

## 6. 比較ログ

各プロジェクト終了時に、以下の形式で比較ログを残す。

```md
## <order>. <project>

### Baseline Impression

### What Carried Over

### What Did Not Carry Over

### Library-Specific Friction

### Bugs or Detours

### Next Implementation Notes
```

比較ログは、各プロジェクトの `IMPLEMENTATION_NOTES.md` に書く。全体まとめが必要になった段階で、`docs` に最終比較レポートを追加する。

## 7. 学習効果の扱い

単一エージェント方式では、後半ほど以下の利点が出る。

- ゲーム仕様を覚えている
- 定数やゲームバランスを把握している
- 衝突処理のパターンを再利用できる
- HUDやゲームオーバーの構造を再利用できる

一方で、以下の不利もあり得る。

- 前のライブラリの設計を無理に持ち込む
- 物理エンジン向きではない手動処理を引きずる
- 3Dエンジンに2Dの設計を押し付ける
- 実装順序によって評価が偏る

最終比較では、点数とは別に「順序による影響」をコメントする。

## 8. 記録すべき時間

厳密なタイムトラッキングは必須ではないが、可能なら以下を記録する。

| 記録 | 意味 |
| --- | --- |
| Start | 実装開始 |
| First Render | 画面表示 |
| First Control | プレイヤー移動 |
| First Combat | 射撃と敵撃破 |
| Core Complete | 必須仕様完了 |
| Verification Complete | 手動確認完了 |

## 9. 最終レポートで見る観点

最終的に、以下の問いに答える。

- 最初に作るならどれが一番楽か
- 2Dアクションを作るならどれが自然か
- 低レベルな実装まで制御しやすいのはどれか
- 物理込みで破綻しにくいのはどれか
- 3Dに広げるならどれが有利か
- AIがAPIを間違えにくいのはどれか
- AIが古い情報に引っ張られやすいのはどれか
- 連続実装で学習効果が最も効いたのはどれか
- 逆に、前実装の知識が邪魔になったのはどれか

## 10. 次の実行ステップ

次に実装を開始する場合は、`phaser` から始める。

Phaser版が完了したら、その実装を基準線として、Excalibur以降では必ず「Phaserと比べて何が楽か、何が難しいか」を記録する。
