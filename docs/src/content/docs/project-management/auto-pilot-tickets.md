## 方針の再検討

### 採用する考え方

危険回避は戦術モードではなく、すべての移動へ適用する不変条件です。現行のMovementはすでに、回避可能な衝突を先に除き、安全包絡内でのみ回収、交戦、周回の目的を比較します。この順序は変更しません。

事前計測で既存`patrol`が0.069%しかないと判明したため、改善対象を平穏・低圧時の期限付き巡回候補へ改訂しました。現在の周期方向は訪問場所を記憶しないため、一定時間訪れていない安全領域へ向かう長期目標をopt-inで追加します。

安全な高価値XP、レベル直前、短いETA、地上XP数に応じた連続回収圧、回復、交戦は残します。直接XP追跡の全面廃止と、常時非停止は行いません。

### そのまま採用しない点

四隅の座標へ順番に強制移動する方式は採用しません。

- 角は逃走方向が少なく、敵弾や接触敵が多いと意図的に不利な場所へ入ります。
- 障害物、アリーナ崩壊、将来のステージ形状により、固定座標が到達不能になります。
- 固定順路は敵出現との位相が揃い、特定武器だけに有利な比較器になる可能性があります。
- 停止が最安全となる弾幕や狭路があるため、非ゼロ移動を強制すると安全包絡と矛盾します。

現在の安全領域を3 x 3の論理領域へ分け、各領域内の到達可能な内側地点を候補にします。未訪問時間を主因、経路ETA、経路危険、逃走余地を補助因子として選びます。危険で到達できない領域は負債を残したまま一時保留し、安全になった後で再評価します。

## 戦術規則

領域巡回は危険フェーズを上書きする最上位Intentではなく、平穏・低圧時だけ既存候補と比較する期限付き`patrol`です。

| 状態 | 既存Policy | 巡回の扱い |
| --- | --- | --- |
| `pending` | 回復、XP、交戦、`patrol`のUtilityを比較する | 15秒から30秒の領域年齢に応じて巡回候補を比較する |
| `warning` | 緊急回復を除き、安全なXP密集回収を優先する | 巡回しない。巡回時計も進めない |
| `active` | 巡回と意図的XP回収を外し、回避しながら敵数を減らす | 巡回しない。巡回時計も進めない |
| `recovery` | 連続`backlogPressure`に応じて回収範囲と価値を上げ、通常Utilityへ戻る | 安全な直接経路がない場合はXPを含む未訪問領域を候補にする |

危険中に実際に通過した領域は訪問済みとして記録してよい一方、`warning`と`active`の時間は領域年齢へ加算しません。イベント終了直後に「期限超過した角へ直行する」副作用を防ぎます。

`fair`と`ceiling`は同じ領域分割、訪問時計、経路危険上限、コミット時間を使います。`ceiling`の既存武器別戦術と直接回収は維持し、巡回だけで技能上限を置き換えません。可視情報と反応遅延だけを使う`human-reference`は別の将来課題です。

## 領域モデルの初期仮説

初期値は採用値ではなく、感度分析する仮説です。

- 現在の崩壊内側矩形を3 x 3へ分割する。
- 各領域の中心と決定論的な代替点から、障害物余白と到達可能性を満たす地点を選ぶ。
- 訪問半径は領域短辺から算出し、72pxから80pxへ制限する。
- 長期計画は毎フレーム再計算せず、通常1Hz、目標到達と地形変更時だけ即時更新する。
- 目標は到達、到達不能、領域消失、0.9秒の進捗停止で解除できる。
- 巡回目標はAgent内で維持するが、別Intent、直近衝突、イベント遷移は即時中断できる。
- 到達不能領域は短いクールダウンを設け、同じ地点への再固執を防ぐ。
- 領域選択の係数は`fair`と`ceiling`、PulseとSpreadで共通にする。

領域数や秒数を固定する前に、50%増減した候補で結果が反転しないことを確認します。

## PH-AUTO-001 巡回モデル・比較スイッチ・基準計測

目的:

現行入力を変えず、訪問履歴モデルとBaseline計測を追加します。実ランで`patrol`の空き時間が十分に存在するかを先に確認し、効果が見込めない実装を広げません。

対象範囲:

- `autoPilotCoverage.ts`へ領域生成、訪問判定、候補評価、目標記憶を分離する。
- `AutoPilotAgent`が訪問時刻、現在目標、開始時刻、進捗、保留期限を所有し、`reset()`で消去する。
- 現在の崩壊内側矩形から領域を作り、障害物内、領域外、到達不能な地点を除外する。
- `patrolStrategy: periodic-v3 | visit-history-v1`を追加し、既定を`periodic-v3`にする。
- Probeへ戦略ID、到達可能領域数、30秒 / 120秒窓の訪問率、巡回Intent率、再訪間隔を追加する。
- 現行v3の固定シードBaselineを同じ指標で取得する。

対象外:

- 既定戦略と`InputSnapshot`列の変更。
- 危険イベント、回収、交戦の優先順位変更。
- ゲーム本体のPickup、磁力、障害物、敵出現ルール変更。
- 新しい経路探索ライブラリの導入。

受け入れ条件:

- 既定`periodic-v3`で、変更前と同じシードのDecision / Inputハッシュが一致する。
- 同じWorldState、訪問記憶、シードから同じ領域ID、目標点、Utility内訳を返す。
- 四隅の画面端そのものを目標にせず、プレイヤー半径と安全余白を確保する。
- 障害物で塞がれた地点は領域内の代替点へ移り、代替がなければ一時保留する。
- 崩壊Insetの変更後、領域と訪問判定を現在の安全矩形へ再配置する。
- 武器種と`fair` / `ceiling`を変えても、同一危険状態の領域係数は変わらない。
- 純粋関数、`reset()`、比較スイッチ、Baseline計測の試験がある。

事前停止条件:

- 開発24シード x 2武器の48ラン中、1秒以上の`patrol`を持つランが12本未満。
- または、全Decisionに占める`patrol` Intentが1%未満。

どちらかに該当した場合、`patrol`置換だけでは地上XPへ十分に影響しないため、`PH-AUTO-002`へ進まずPolicyの優先関係を再設計します。

実測結果（2026-07-15、開発24シード x Pulse / Spread、最大450秒）:

| 指標 | 結果 | 判定 |
| --- | ---: | --- |
| 1秒以上の`patrol`を持つラン | 0 / 48 | 停止条件に該当 |
| 全Intentに占める`patrol` | 0.069% | 停止条件に該当 |
| 30秒窓の平均領域訪問率 | 56.4% | 参考値 |
| 平均地上XP | 115.7個 | 50個目標を超過 |
| 最大地上XP | 336個 | 後半の滞留を確認 |

この結果により単純置換案は不採用とし、`PH-AUTO-002`を下記の期限付き候補方式へ改訂しました。既定`periodic-v3`の入力は、領域テレメトリ有無で同一WorldStateまで一致する回帰試験を追加済みです。

## PH-AUTO-002 訪問履歴巡回のopt-in統合

目的:

`visit-history-v1`だけ、最終訪問から15秒でUtilityを上げ始め、30秒で最大になる到達可能領域を平穏・低圧時の巡回候補として既存Utility比較へ加えます。単なる`patrol`置換ではなく、射撃を継続しながら安全包絡内で広い場所へ移動します。

危険予告中のXP、危険本番の交戦・生存、緊急時、XP backlogは巡回より先に確定し、既存フェーズ戦術を変更しません。

対象範囲:

- `visit-history-v1`選択時の`patrol`へ領域ID、目標位置、経路方向を設定する。
- 訪問から15秒から30秒の年齢と危険度から巡回Utilityを決め、危険度0.2までは低圧、0.55以上では追加Utilityを0にする。
- 連続`backlogPressure`を持つ安全なXPは巡回より先に比較する。
- 安全な直接XP経路がない場合だけ、古さとXP密度を合わせた領域を候補にする。
- 実際に領域へ入った場合だけ訪問時刻を更新する。
- 到達、消失、到達不能、進捗停止で巡回目標を更新する。
- 別Intent、直近衝突、`warning` / `active`への移行で即時中断する。
- `warning` / `active`中は巡回時計を止め、通過した領域だけを訪問済みにする。
- 毎フレーム全領域を経路探索せず、更新間隔と候補上限を守る。

原則対象外:

- `autoPilotMovement.ts`、`autoPilotThreat.ts`、`autoPilotTargeting.ts`、`navigationField.ts`。
- HUD、`WorldState`、保存スキーマ、通常ランJSON。
- 危険予告のXP、高密度回収、緊急回復、危険本番の交戦・生存規則。
- 敵AI、武器数値、危険イベント時間の調整。

対象外の変更が必要になった場合は、このチケットへ混ぜず要件を再検討します。

受け入れ条件:

- 無危険の固定配置で、30秒以内に到達可能領域の8 / 9以上を訪れる。
- 角領域が危険または到達不能な場合、同じ目標へ固執せず別の安全領域を選ぶ。
- `warning` / `active`、緊急時、backlogのIntent優先順位は`periodic-v3`から変わらない。
- `warning` / `active`中の巡回Intentは0で、既存の回収・交戦戦術を維持する。
- 停止が最安全な候補を残し、非ゼロ移動を強制しない。
- 回避理由なしの巡回連続停止は1秒未満を維持する。
- 武器とプロファイル間で領域係数が一致する。
- 決定論、目標維持、`reset()`、障害物代替、崩壊無効化、危険時上書きの試験がある。

主な対象:

- `phaser/src/simulation/autoPilotCoverage.ts`
- `phaser/src/simulation/autoPilotPolicy.ts`
- `phaser/src/simulation/autoPilot.ts`
- `phaser/src/simulation/autoPilot.test.ts`
- `phaser/src/simulation/autoPilotPolicy.test.ts`

## PH-AUTO-003 凍結比較と採否判定

目的:

現行`periodic-v3`と`visit-history-v1`を同じ乱数入力で比較し、XP、生存、公平性、無謀回収、無駄回復、性能を含めて採用、調整、削除を決めます。

追加計測:

- 巡回目標の選択、到達、放棄、一時保留と理由。
- 到達可能領域数、30秒窓訪問率、最古領域年齢、巡回ETA。
- 巡回Intentに対するMovementの回避上書きと、理由なし停止。
- 既存の地上XP、XP / 分、強化到達、無謀回収、無駄回復、フェーズ別Intent、p95。
- 全候補を分母とする安全回収機会率。

実験順:

1. 手書きマイクロケースで領域、崩壊、障害物、危険遷移を確認する。
2. `periodic-v3`と`visit-history-v1`を開発24シード、Pulse / Spread、最大450秒で比較する。
3. 初期定数を50%増減し、少数の値だけで結果が反転しないことを確認する。
4. 定数を凍結し、未使用64シード、最大650秒でホールドアウトする。
5. ブラウザ観戦で不自然な角往復、停止、表示との矛盾を確認する。

初期合格基準:

| 指標 | 合格基準 |
| --- | --- |
| ハード危険違反 | 回避可能な直近衝突への進入0件 |
| 無危険時の領域到達 | 30秒で到達可能領域の8 / 9以上 |
| 実ランの巡回 | 非`active`期の30秒窓訪問率75%以上、再訪間隔p95 30秒以下 |
| 地上XP | 非`active`期の50個超過率がBaseline以下。目標5%以下 |
| XP進行 | XP / 分と強化到達時刻がBaseline比-5%以内 |
| 停止 | 回避理由なしの巡回連続停止1秒未満 |
| 無謀回収 | Baseline比+5ポイント以内 |
| 無駄回復 | 回避可能な満HP取得をBaselineより増やさない |
| 生存非劣性 | 650秒RMST差の95%区間下限が-13秒より大きい |
| fairの回収差 | Pulse / Spread間のXP / 分差をBaselineから5ポイント以上広げない |
| 決定論 | 同一シード・設定のDecision / Inputハッシュが完全一致する |
| AI性能 | 後半状態の判断p95 1.5ms以下 |

採用条件:

- 上記を満たし、Pulse / Spreadのどちらか一方だけが領域巡回で恒常的に有利にならない。
- 観戦上、同じ角への往復や危険な固定順路が目立たない。
- 採用時だけ既定を`visit-history-v1`へ変え、`AUTO_PILOT_MODIFIER_ID`を`tactical-observer-v4`へ更新する。
- AIランの通常ランキング除外と、`O`による手動切替をE2Eで再確認する。
- ゲーム本体の`rulesetVersion`と通常ラン記録を変更しない。
- 結果を設計判断、現在地、意思決定記録へ反映する。

停止・差し戻し条件:

- 接触被ダメージ、無謀回収、満HP回復取得のいずれかが一貫して悪化する。
- 崩壊後の消失領域を追い続ける、到達不能目標で停止する。
- 特定武器だけが改善し、fairの差が広がる。
- AI判断p95を守れず、候補制限でも回復しない。
- 少数の定数だけで結果が反転し、開発セット外で改善を再現できない。

不合格時は既定を`periodic-v3`のまま維持し、opt-in実装を削除するか検証専用に残すかを明示します。

## PH-AUTO-003 結果

2026-07-15の開発セットで、単純な訪問履歴候補を24シード x 2武器、最大450秒まで比較しました。

| 指標 | `periodic-v3` | 初回`visit-history-v1` | 差 |
| --- | ---: | ---: | ---: |
| 平均生存 | 445.2秒 | 448.9秒 | +3.7秒 |
| 30秒窓訪問率 | 69.0% | 70.7% | +1.7ポイント |
| 平均地上XP | 115.6個 | 119.2個 | +3.6個、悪化 |
| 非activeの50超過率 | 70.6% | 71.4% | +0.8ポイント、悪化 |
| XP / 分 | 378.7 | 380.9 | +0.6% |
| 敵弾被ダメージ | 301.6 | 331.4 | +9.9%、悪化 |
| 回避可能な満HP回復 | 3786件 | 3992件 | +206件、悪化 |

この版は停止条件に該当したため不採用としました。その後、50個超で安全な直接経路がない場合だけXP密度を使い、15秒から30秒で巡回Utilityを上げる候補へ修正しました。代表6シードでは最大地上XP299→278、非active超過率70.7%→69.8%、生存+7.9秒まで改善しましたが、30秒窓訪問率73.0%で75%へ届かず、敵弾被ダメージも280.7→297.3へ増えました。

結論:

- 既定は`periodic-v3`とし、`AUTO_PILOT_MODIFIER_ID`はv3から変更しない。
- `visit-history-v1`は検証専用opt-inとして残し、実験ランへ`auto-pilot:patrol:visit-history-v1`を付ける。
- 180秒単独性能は判断p95 1.424ms、coverage p95 0.031msで予算内だった。
- 開発サブセットで採用条件を満たさないため、残り18シードと64シードホールドアウトは実施しない。
- 再開条件だったPickup候補分母と経路拒否理由の計装は、第二次監査対応の`PH-AUTO-013`で追加した。次の比較は`PH-AUTO-017`の明示的なXPコリドータスク完成後に行う。

## 第二次監査対応

### PH-AUTO-012 停滞解除と再固執防止

状態: 完了。

- 同一Pickupでも0.35秒経路進捗がなければPolicy評価前にコミットを解除する。
- 解除対象を0.9秒候補から除外し、別候補を探索する。

## EX Protocol candidateのprobe policy

`PH-V08-027`のbalance / exposure probeは、製品AutoPilotを変更せず、probe専用policyで次だけを固定します。

- weaponとseedに対するProtocol、E1、E2 route。
- Reboundの次volley武装。
- Tidal / Breakwaterのchargeがあり、前方clusterへ意味がある場合のspecial。
- 全Protocol共通の既存fair移動、回避、回収、照準。

RelayとRedlineはfair AutoPilotの自然照準で成立機会が不足する場合があります。0回を即座に弱いProtocolと判定せず、`opportunity=0`と`opportunity>0 / effect=0`を分けます。probe policyを強くするために製品AIの生存方針、武器別安全係数、ランキング適格性を変更しません。

このprobeは20 seedのreview triggerです。人間のspecial timing、代償理解、武器弱点維持を合格判定しません。結果を見てcandidate値を自動調整せず、変更案は別比較として事前登録します。
