## 優先度1

### 長時間ランの成長停滞と永久生存

状態: v0.7 RC5でC5到達性を維持。600秒以降は公開ベータで継続観測。

v0.5ではXP必要量が指数的に増え、レベル20以降に成長が終わったように見えていました。

対応:

- `PH-V06-001` で体験上の選択間隔からXP曲線を再設計する。
- 最初の強化、最終強化、選択が空いた最長時間を記録する。
- ビルドの主軸完成後は、通常強化と分離したEX進行へ移す。
- 敵の能力はEX成長より速く上げ、画面密度側には安全上限を置く。
- 反復危険イベントと600秒以降の戦場収縮で安定周回を崩す。

実績: `growth: 1.25`、上限120、`buildCompletedAt`に加え、循環型EX、無制限の脅威ティア、3イベントのシャッフルバッグ、アリーナ崩壊を実装しました。RC4でEndlessのEX曲線を180 / 1.04 / 360へ緩和し、固定6ランのC5以上を0 / 6から3 / 6へ改善しました。RC5はこの曲線を維持します。平均生存は3.8%増に留まり、全6ランが900秒前に終了したため永久生存は再発していません。最終遠征はStage固有曲線を維持します。600秒以降の崩壊体感は公開ベータで観測します。

### 武器差と強化選択の観測不足

状態: v0.6.5で軽減済み。

v0.5の手動5ランはすべてPulseで、全ランが`splitShot`ランク2、`vitalCore`ランク0でした。ただし、現在の記録だけでは「候補へ出なかった」と「提示されたが選ばなかった」を区別できません。射撃数と命中数だけでは、分裂や貫通が狙い方を変えたかも判断できません。

対応:

- `PH-V06-008`で強化の提示、選択、除外理由、時刻を記録する。
- 1射あたりの異なる命中敵数、敵種別撃破、移動距離を開発計測へ追加する。
- PulseとSpreadを固定条件で比較し、スコア以外の差を採否根拠にする。
- プレイヤー履歴へ詳細イベントを無制限に保存せず、要約と開発ログを分ける。

実績: v0.6.3でPulseの分裂を集束共鳴へ置き換え、Spreadへ掃射循環を追加しました。v0.6.4ではPulseだけ最大約1405px/sへ伸ばし、連続衝突、貫通順、外周反射を追加しました。固定10シードの主要差は6%未満、手動Pulseは26355点 / 353.96秒まで伸び、通常ビルド完成とEX Lv3へ到達しました。v0.6.5では反射後命中・撃破を障害物と外周へ帰属できるようにしました。

### 弾と無敵時間の仕様不一致

状態: v0.6で解消。

総命中可能数を`hitCapacity`へ統一しました。無敵時間中に重なった敵弾は、ダメージを与えず消滅する意図的なルールです。

対応:

- `PH-V06-010`で貫通の用語、偶数分裂時の配置、敵弾消費ルールを固定する。
- 分裂と貫通の組み合わせを固定配置テストで確認する。
- 意図的な仕様差は文書化し、挙動変更時は`rulesetVersion`を更新する。

実績: 1、2、3体への総命中数、偶数分裂の中央空き、同じ敵への再命中禁止、敵弾消費を固定配置試験で確認しました。

### 乱数列の連動

状態: v0.6で解消。

敵出現と強化候補が同じ乱数列を使っています。新しいドロップや戦闘展開抽選を追加すると、同じシードの強化候補まで変化します。

対応:

- `PH-V06-007` で用途別乱数列へ分ける。
- 乱数列の派生規則と相互非干渉を単体テストする。

実績: `arena-rng-v1`で出現、強化、ドロップ、戦闘展開、ステージ候補を分離し、無関係な抽選追加で列が変わらないことを単体試験しました。

### ルールセットとランキングの整合性

状態: RC6で実装済み。旧記録読込と比較意味論はfixtureで固定した。統合QA完了までproduction昇格は保留。

バランス変更後も同じ設定版を使うと、比較できないランが同じランキングへ混ざります。開発ログには、デバッグで強制終了した0秒ランも含まれます。

対応:

- v0.5のラン記録へ `rulesetVersion`、`runOrigin`、`rankEligibility` を追加する。
- デバッグ、自動テスト、固定シードの区分を明示する。
- ゲームルール変更時の版更新をテストまたは設定指紋で強制する。

実績: `appVersion`と`rulesetVersion`を分離し、手動 / デバッグ / 自動テスト、random / fixed seedを記録しました。デバッグと自動テストは通常ランキングへ入りません。

RC6対応:

- Expeditionの主記録を勝利ランの総クリア時間へ統一する。
- 敗北は履歴へ残すが勝利PBを上書きしない。
- `comparisonScope: overall | weapon`を問い合わせ側で選び、総合PBと武器別PBを両立させる。
- fixed runは実seed値ごと、random runはrandom同士で比較する。
- 速攻成果を時間メダルへ移し、戦術点との二重評価を止める。
- rulesetを`phaser-v0.7.0-final-expedition-rc6`へ更新し、RC5以前をRC6順位へ混ぜない。

実績: 勝利時間優先、敗北除外、overall / weapon、fixed実seed、RC5 / RC6、時間非加点を保存・比較fixtureで保証しました。ランキング保存はoverallと各weapon上位の和集合とし、履歴上限後も双方のPBを保持します。

停止条件: 敗北による勝利PB上書き、fixed seed間の混在、overallとweapon PBの排他、旧rulesetの混入が1件でも再現した場合はproductionへ進みません。

### Encounter時計と構造化出現のsoft lock

状態: RC6で対応予定。

Act表示だけをrun時間から補正すると、Encounter Directorが次Actを開始したままHUDだけが前Actを示す可能性があります。また、Commanderを配置できない状態で無期限に再試行すると、Act停止と後続カード停止が解除されません。

対応:

- `runElapsed`はRun Lifecycle / World、`actElapsed`はEncounter Directorが所有する。
- HUDはDirectorの現在Actを表示するだけにする。
- Commander cardはAct時計を明示的に止める。
- 120秒はspawn成功後のactive時間だけで数える。
- spawn deferは決定論的な間隔と配置期限を持ち、期限切れを`deployment-timeout`として記録する。
- 撃破、active timeout、deployment timeoutの全経路でAct時計を一度だけ解放する。

停止条件: Act境界をCommander active中に越える、spawn前に120秒を消費する、配置不能で後続進行が止まる、同一seedの再試行結果が変わる場合はコンテンツ追加を止めます。

### ローカル保存の破損と重複記録

状態: v0.5で軽減済み。

`localStorage` は手軽ですが、途中書込、手動編集、古い形式、容量超過によって読込や保存が失敗します。また、ゲームオーバー後の再描画や画面遷移で同じランを複数回保存する危険があります。

対応:

- `PH-V05-001`で保存形式へ版を付け、読込時に構造検証する。
- 破損項目だけを無視または退避し、ゲーム全体を起動不能にしない。
- ランIDと終了処理の所有者を固定し、同じランの重複保存を単体テストする。
- 保存失敗をリザルト表示とゲーム進行から切り離す。
- `PH-V05-011`で破損、件数上限、容量超過相当を確認する。

実績: Zod検証、破損退避、旧形式移行、ID upsert、終了処理の一回性、容量超過時のリザルト継続を単体テストとE2Eで確認しました。

### 障害物と弾の関係

状態: v0.6.5で軽減、ベータ観測へ移行。

障害物が移動妨害に偏り、敵が正面へ張り付くことで安全解になりやすい状態でした。

対応:

- `PH-V04-005` の初回対応として中央障害物を削除した。
- 跳弾のデータと処理は追加済み。
- Pulse最終強化として1回跳弾を実装した。
- Pulse最終強化を固定外周へ拡張し、敵出現を塞がない一方向反射フィールドとした。
- 敵は射線が切れた場合に共有経路場で迂回し、射撃体は遮蔽物越しに撃たない。

残存リスク: 同じ経路へ敵が集中する見た目と角での細かな方向転換は残ります。外周反射の成果はv0.6.5以降の障害物 / 外周別追撃で観測します。

### 操作負荷

状態: v0.5で設定可能。

`WASD + マウス照準 + 射撃維持` の同時操作負荷が高い状態です。

対応:

- 自動射撃をv0.5の既定操作として暫定採用する。
- `PH-V05-010`で自動射撃を切り替え可能にする。
- 左クリックとSpaceを手動射撃として維持する。
- 防御ダッシュと右クリック固有スキルはv0.6以降へ延期する。

実績: 自動射撃は既定で有効、設定から無効化可能です。無効時も左クリックとSpaceで射撃でき、右クリックは射撃へ反応しません。

## 優先度2

### 観戦AIの巡回が武器比較を歪める

状態: opt-in実装と開発比較を完了、既定採用見送り。

訪問履歴を持たない現在の周回はフィールド端のXPを残しやすい一方、固定された四隅を順番に回る方式は、逃走方向の少ない場所へ強制的に入り、敵出現との位相や武器射程によって比較結果を歪めます。崩壊後の消失領域や障害物内を追うと停止も再発します。

対応:

- `PH-AUTO-001`で現在の安全矩形を3 x 3へ分け、領域内の到達可能な内側地点を選ぶ。
- 未訪問時間を主因とし、XP密度、経路ETA、経路危険を補助因子にする。
- 安全な高価値XP、レベル直前、短いETA、35個超の密集回収を維持し、50個超で安全経路がない場合だけXP密度領域を使う。
- 危険回避を全Intent共通の安全包絡として維持し、危険本番は巡回を中断する。
- 危険予告と本番中は巡回時計を止め、イベント直後の角への固執を防ぐ。
- 巡回係数を武器と`fair` / `ceiling`で共通化し、武器固有差を照準と射撃機会へ限定する。
- Baselineとのペアシード比較で、生存、無謀回収、無駄回復、領域訪問率、地上XP、p95を同時に確認する。

停止条件: 到達不能目標での停止、接触被ダメージの一貫した悪化、特定武器だけの恒常的改善、AI判断p95 1.5ms超過が候補制限でも解消しない場合は採用しません。

実績: 24シードの初回候補は訪問率と生存を改善しましたが、地上XP、敵弾被ダメージ、満HP回復が悪化しました。密度fallback後の6シードも訪問率73.0%で基準75%へ届かず、敵弾被ダメージが増えたため、既定`periodic-v3`を維持しました。候補は明示modifier付きの検証専用opt-inとして分離し、公開ベータとv0.7へ影響させません。

### ArenaSceneへの責務集中

状態: 表示・メニュー分割済み、session / debug分割が残存。

現在の `ArenaScene` はラン生成、デバッグAPI、画面遷移、ログ送信を調停しています。保存、ランキング、プロフィール、複数画面をそのまま追加すると、ラン終了処理の重複とテスト困難が起きます。

対応:

- ラン記録生成、順位計算、保存復旧をPhaser非依存モジュールへ分ける。
- リザルトとタイトルは責務が増えた時点で専用クラスへ分ける。
- `ArenaScene`はラン開始、終了、画面間の調停だけを所有する。
- 将来だけの空クラスは作らず、各チケットで必要になった責務だけを抽出する。

実績: ラン記録、順位、保存、プロフィール、BGM、フィードバックに加え、表示文を純粋Presenter、ワールドと画面を個別View、メニュー状態をControllerへ分離しました。`PhaserArenaRenderer`は935行から51行、`ArenaScene`は1274行から1129行になり、全Playwright機能・画像回帰を通過しています。次はsession / run lifecycleとdebug / AI / performanceを分け、Sceneの1000行超を解消します。

### 公開ルールとルール版の不一致

状態: v0.6.8公開版で解消。

Pulse外周反射を公開採用し、`RULESET_VERSION`を`phaser-v0.6.8-pulse-boundary-ricochet`へ更新しました。旧ルール記録は履歴へ残り、ランキングは完全一致する比較キーだけを選ぶため、新旧記録は混ざりません。

対応:

- `PH-BETA-003`へ外周反射の有効採用と手動4ランの根拠を記録する。
- 設定、表示、ルール版、ランキング比較キーの一致をproductionで再確認する。
- 変更前後の記録を自動移行で混ぜず、別条件として残す。
- 公開ベータ基準コミットとCloudflare Version IDを記録する。

### 実験ブランチと公開基準の混在

状態: 公開基準commitを固定。v0.7ブランチ分離が残る。

`experiment/auto-pilot-observer`上の公開ベータ基準をcommit `ff686f992a65`として固定し、Cloudflareへ同じSHAを配信しました。ここから直接v0.7のルール変更を重ねると、公開ベータの修正と実験入力の境界が再び曖昧になるため、次の製品変更は別ブランチへ分けます。

対応:

- 既存変更を破棄せず、意図と検証結果を確認してコミット単位を決める。
- 公開ベータ基準を`main`または明示したrelease branchへ固定する。
- v0.7はその基準から別ブランチで開始する。
- 観戦AIのopt-in結果を製品ルールの採否へ直接混ぜない。

### v0.6自動入力と人間操作の差

状態: 手動採否済み、継続観測。

固定10シード比較では集束、掃射、反響回路、複数標的数に武器差が出ましたが、`kiteCollect`は人間の学習、意図的な標的維持、射角作り、障害物利用、危険予告の理解を再現しません。

対応:

- PulseとSpreadを各2本以上手動プレイする。
- 自動比較は回帰と役割差の信号に限定し、面白さの完了条件にしない。
- 反響回路、射撃体集中、契約は機能フラグで個別に無効化できる状態を維持する。
- 手動評価前に数値やコンテンツを追加しない。

実績: Pulse 7972点 / 163.03秒と26355点 / 353.96秒、Spread 34434点 / 421.36秒を比較しました。Pulseは操作集中で通常ビルド完成とEXへ到達できたため採用し、自動入力は引き続き回帰用途だけに使います。

### v0.7自動到達性と人間採否の混同

状態: 技術preview採用、production昇格保留。

RC5の3 seed x 2武器はすべて8分50秒から9分26秒で指揮艦を撃破し、決定性、密度、制圧衝撃波の可避fixtureも通過しました。ただし観戦AIは予告理解、入力負荷、学習曲線、再挑戦意欲を人間と同じ条件で評価しません。

対応:

- 自動6ランは到達不能、再現性、上限逸脱の検出だけに使う。
- v0.7をproductionへ昇格せず、Version Preview URLへ限定する。
- RC3 / RC4の通常UI 5勝とRC5自動6勝は基準証跡に使い、RC6でPulse / Spread各1本以上の中央周回、予告理解、回復経路を再確認する。
- 単一高得点や自動クリア時間だけで武器、敵、カード、ボス数値を変更しない。

### ボス周回と回復供給の自己強化

状態: RC5で局所対策済みだが、production採用は未確定。RC6で有限回復候補を独立比較する。

RC3 / RC4ではボスを中央へ寄せて周回すると、貫通で通常敵を倒しながら回復を継続取得できました。通常UIの5勝では回復が総被ダメージの97.1%から99.5%を相殺し、遅いボス撃破ほど雑魚得点が増えるため、危険を取って速く倒す理由も弱くなっていました。

対応:

- ボス戦中の雑魚由来回復を決定論的に毎秒1個までとし、生成数と抑制数を記録する。
- 範囲外退避か障害物遮蔽を要求する`command-pulse`を追加し、近距離周回へ周期的な位置変更を要求する。
- RC5では完遂15,000点とボス戦240秒基準の速攻ボーナスを追加した。RC6では総クリア時間を主記録にするため、速攻成果を時間メダルへ移して戦術点から外す。
- 回復全体やEndlessをナーフせず、最終遠征のボス戦だけへ対策を閉じる。
- 通常UIで従来周回を意図的に再試行し、無期限安定と予告の不可読がないことを確認する。
- RC6基礎版をcontrolにし、必要な場合だけ最終決戦へ有限repair budgetを追加する。
- 回復相殺率を`ボス中の実HP回復量 / ボス中の総被ダメージ`と定義し、総被ダメージ0は算出不能とする。

停止条件: repair budget候補が道中またはEndlessへ波及する、片方の武器だけを到達不能にする、drop数だけを減らして回収判断を消す場合は採用しません。

### 公開ベータの性能と診断情報

状態: 観測基盤、実GPU耐久、公開スモークを完了。

快適という単一端末の所感だけでは、別ブラウザや長時間ランのフレーム劣化を判別できません。一方、汎用のWeb表示指標を増やしてもゲームループのフレーム間隔は直接説明できません。

対応:

- Phaser `TimeStep.rawDelta`から平均、p95、最大、50ms超過数をラン単位で保存する。
- Phaser `actualFps`と、外部の1秒`requestAnimationFrame`計測を15分ソークで照合する。
- 敵、弾、ピックアップ、ヒープ、Canvas、コンソールエラーの既存上限を維持する。
- 長時間タスクAPI、Web Vitals、外部監視SDKは、公開ベータで実際に不足が出るまで導入しない。

実績: Phaser 4 productionの15分ソークは58583フレーム、平均15.42ms、p95 29ms、50ms超過4件、Phaser実測32.35fpsでした。最大JSヒープ155.8MiB、最大敵86、全弾257、Canvas非空、コンソールエラー0件です。公開URLの通常UIスモークでもページ例外、失敗リクエスト、HTTP 4xx / 5xxは0件で、現時点の性能リスクは低いと判断します。

### 音源ライセンスとブラウザ自動再生

状態: v0.5で軽減済み。

無料素材でも、再配布、加工、クレジット表記の条件が異なります。また、ブラウザはユーザー操作前の音声再生を制限します。

対応:

- `PH-V05-007`で素材ごとの出典、ライセンス、加工可否を資産台帳へ残す。
- 初回ユーザー操作後に音声を開始する。
- BGMの二重再生、画面遷移、ミュート復元をテストする。
- 音源が読めない場合もゲームを継続できるようにする。

実績: 外部素材を使わず、プロジェクト内で再生成できる音源と台帳を`phaser/public/audio/README.md`へ追加しました。手動所感で単調さが指摘されたため、高頻度SEへ2〜3バリエーション、BGMへ4区間・32秒の展開を追加しました。16音源の読込、音声ロック待ち、画面遷移、ミュート、再読込をテストしています。外部曲の追加選定はライセンス確認を伴う任意改善として残します。

### 浅いゲームループの過剰な装飾

タイトル、BGM、ランキングだけを磨き続けると、見た目は良くなってもランごとの差が増えません。

対応:

- v0.5は記録と画面の完了条件で閉じる。
- v0.6は開始武器、成長間隔、最終強化を体験可能な成果物にする。
- 表示改善をゲームプレイ仮説の代替にしない。

### バンドルサイズ警告

Phaser / Viteのビルドでバンドルサイズ警告が出ます。

対応:

- ビルド成功時は既知警告として扱う。
- リリース準備または実測性能に問題が出た時点で `PH-V03-009` を再開する。

### 開発サーバー依存の低重要度警告

`npm audit`は、Viteが使う`esbuild 0.27.7`について、Windows上の開発サーバーで任意ファイルを読まれる可能性がある低重要度警告を1件報告します。現在の開発・配信環境はKubuntuと静的ビルドであり、公開サーバーへVite開発サーバーを使いません。

対応:

- 高・重大の警告がないことを依存更新時に確認する。
- `npm audit fix`で未検証の依存更新を自動適用せず、Viteが対応版を取り込んだ時点で通常更新する。
- 開発サーバーを外部公開しない。

### ピックアップ取得演出

取得演出はまだ最低限です。

対応:

- 他の重要フィードバックと合わせ、`PH-V03-006` で小さく対応する。

## 後回し

### アイテム拡張

アイテムは有力ですが、操作、開始武器、強化、戦闘展開が固まる前に増やすと、調整原因が分かりにくくなります。

対応:

- 最初の技術試作は `haste` 候補とする。
- 本格的な候補追加はv0.7以降に再評価する。
