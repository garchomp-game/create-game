## 目標

`Arena Core` を、現在のプレイ可能な試作から「ビルドを変えて何度も挑戦できるローカルゲーム」へ進めます。

技術基盤だけを連続して作らず、各バージョンにプレイヤーが体験できる変化を最低1つ含めます。

体験要件は [ゲームプレイ拡張設計](../../design/gameplay-expansion-blueprint/) を正本とします。

## 作業順を変更した理由

以前はv0.6でステージ基盤、v0.7で装備を想定していました。プレイログを再確認すると、長時間ランの主要問題はステージ不足だけではありません。

- 約14.7分のランでもレベル20、強化19回で、次のレベルに1792 XP必要だった。
- 全強化ランク24へ届くには、現行式で累積約24088 XP必要になる。
- 90秒以降は全敵種が揃い、敵構成の重みが固定される。
- 通常ランの開始武器が `pulse` に固定されている。

そのため、ステージ追加の前にv0.6でビルドの個性とエンドレス後半を扱います。同じアリーナでも別のランを試したくなる状態を先に作ることで、後続ステージの価値を正しく評価できます。

## 進め方

- 1チケットで1つのプレイヤー体験仮説を検証する。
- ゲームプレイチケットには計測と固定シード状態を含める。
- 試作は設定で無効化または削除しやすい形を優先する。
- コンテンツ定義と基盤変更を1チケットへ大量に混ぜない。
- 各バージョン終了時に、短時間、通常、長時間の手動ランを記録する。
- 完了条件を満たさない試作は、採用だけでなく削除も選べるようにする。

## 相対見積もり

1サイクルを「限定した実装、自動検証、手動ラン3本、採用判断」とします。実日数は作業速度が固まってから見積もります。

| 節目 | 想定サイクル | 主な不確実性 |
| --- | ---: | --- |
| v0.4完了 | 1から2 | ダッシュと入力役割 |
| v0.5 | 3から5 | Canvas上のリザルト、履歴、保存移行 |
| v0.6 | 5から8 | 成長間隔、武器バランス、最終強化の価値 |
| v0.7 | 6から10 | 戦闘展開制御、出現安全、ボスの読みやすさ |
| v0.8 | 4から7 | 解放間隔、チャレンジの再挑戦価値 |
| v0.9からv1.0 | 5から9 | コンテンツ調整、アクセシビリティ、長時間動作 |

v0.6とv0.7が最大の設計リスクです。そこで面白さを確認できなければ、v0.8以降のコンテンツ制作へ進まず設計を戻します。

## 着手前の関門

### v0.4の操作方針

- 自動射撃を標準化するか、設定化するか。
- Spaceキーの防御ダッシュを採用するか。
- 右クリック固有スキルを後続へ送るか。
- 跳弾を武器最終強化へ置くか。

### v0.5の記録基盤

v0.6へ進む前に、次を満たします。

- 手動ラン3本以上が履歴へ残る。
- 自己記録、シード、ビルド、死因をリザルトで確認できる。
- デバッグと自動テストを通常ランキングから除外できる。
- ゲームルール版が異なる記録を分けられる。

詳細は [v0.5作業計画](../endless-polish-plan/) を参照してください。

## v0.6 ビルドの個性とエンドレス後半

個別チケットの最新要件、依存、検証方法は[v0.6チケット詳細](../v06-tickets/)を正本とします。

目的:

- 同じアリーナでも、開始武器と強化判断でプレイを変える。
- 長時間ランで成長選択が止まったように見える問題を解消する。
- エンドレスへ最初の危険と報酬の選択を入れる。

### PH-V06-001 XP曲線と強化選択間隔の再設計

成果物:

- 区間別または上限付きのXP曲線。
- 強化選択間隔のシミュレーション。
- 選択までの時間を表す計測値。

完了条件:

- 最初の強化が5秒から10秒に収まる。
- 開始2分までは15秒から30秒、以降は30秒から60秒程度で選択が発生する。
- 挙動を変える重要な選択を90秒以内に提示できる。
- 10分ランで `longestMeaningfulChoiceGap` が90秒を大きく超えない。

### PH-V06-002 開始武器選択の縦切り基盤

成果物:

- 最初は`pulse`と`spread`のラン開始前選択。
- `weaponId` のラン記録とリザルト統合。
- `pierce`を通常選択へ出すための固定配置・役割判定。

完了条件:

- PulseとSpreadで照準距離、位置取り、標的優先順位の少なくとも1つが変わる。
- 同じシードで武器別ランを再現できる。
- 単純な秒間ダメージ差だけで優劣が決まらない。
- 開始武器別の分析と総合ランキングの扱いを混同しない。

### PH-V06-003 強化分類と効果合成

成果物:

- 武器、機動、生存、補助、スキルのタグ。
- 基礎値、開始武器、強化、一時効果の合成規則。
- `gameConfig.ts` から必要な定義だけを分ける最小コンテンツ登録。

完了条件:

- UI、シミュレーション、計測が同じタグを使う。
- 効果適用順を単体テストで固定する。
- Phaser依存なしでビルド要約を生成できる。

### PH-V06-004 最初の武器最終強化試作

成果物:

- `pulse` の跳弾など、挙動を変える最終強化1つ。
- 必要条件の事前表示と取得演出。

完了条件:

- 取得前後で狙い方または地形利用が変わる。
- 弾の見た目とリザルトから取得状態が分かる。
- 手動ラン3本以上で採用、調整、削除を判断する。

### PH-V06-005 強化選択UIとビルド要約

成果物:

- 現在値から取得後値、ランク、分類、最終強化進行の表示。
- 一時停止とリザルトのビルド要約。

完了条件:

- 3秒から5秒で候補の役割を比較できる。
- 最大ランク、取得不可、条件不足を誤解しない。
- キーボードとポインターで操作できる。

### PH-V06-006 予兆付き危険イベントと契約試作

成果物:

- 危険度段階の状態。
- 120秒から180秒に既存敵を使う予兆付き危険イベント1つ。
- 危険イベント成立後に限り、約240秒で危険とスコア倍率を持つ2択契約を1回提示する試作。

完了条件:

- 通常ランでも到達できる時刻に、数値上昇以外の新しい判断を作る。
- 予兆後の移動方向または標的優先の変化を確認できる。
- 危険と報酬を選択前に説明する。
- 契約なしの標準状態を再現できる。
- 契約を含むルールセットをランキングで識別できる。

### PH-V06-007 用途別乱数列の分離

成果物:

- 出現、強化、ドロップ、戦闘展開用の派生乱数列。

完了条件:

- ドロップ抽選を追加しても同じシードの強化候補が変わらない。
- 派生規則を単体テストで固定する。
- ラン出力でルートシードと乱数列版を追跡できる。

### PH-V06-008 ビルド多様性の計測と比較

成果物:

- 開始武器別のバランス回帰テスト条件。
- 強化提示、選択時系列、選択間隔の計測。
- 1射あたりの異なる命中敵数、敵種別撃破、移動距離の開発計測。

完了条件:

- 武器別の生存、毎分スコア、毎分撃破、ダメージ源を比較できる。
- 最終強化の到達率と時刻を確認できる。
- PulseとSpreadを固定条件で各10試行以上比較できる。
- 提示されなかった強化と、提示されたが選ばれなかった強化を区別できる。
- 1つの最適ビルドだけへ収束していないか、手動ログと合わせて判断できる。

### PH-V06-009 Phaser画面とデバッグ責務の分割

成果物:

- タイトル、リザルト、記録、設定のうちv0.6で変更する画面の専用コンポーネントまたはViewModel。
- `window.__ARENA_DEBUG__`とfixtureをScene外へ出す`ArenaDebugBridge`。
- 開発ランPOSTをScene外へ出すクライアント境界。

完了条件:

- `ArenaScene`はフレーム更新、ラン開始 / 終了、画面遷移の調停を主責務とする。
- 順位、保存、プロフィール、画面文言、開発HTTPの具体処理をSceneへ追加しない。
- v0.5の単体テスト、機能E2E、画面比較を変更前後で維持する。
- 行数削減だけを目的にせず、v0.6で実際に変更する責務だけを抽出する。

### PH-V06-010 弾・貫通・無敵時間ルールの監査

成果物:

- 貫通数の表示用語と内部意味の統一。
- 偶数分裂時の弾配置ルール。
- 無敵時間中に重なった敵弾を消費するか残すかの決定。
- 分裂と貫通の固定配置テスト行列。

完了条件:

- 基礎Pulse、貫通+1、Pierceの命中可能数を同じ言葉で説明できる。
- 分裂0、1、2と貫通0、1、2の組み合わせを自動確認できる。
- 敵接触と敵弾の無敵時間ルールを文書化する。
- 挙動を変える場合は`rulesetVersion`を更新する。

### PH-V06-011 v0.6統合プレイテストと採否判定

成果物:

- PulseとSpreadの固定シード比較。
- 武器ごとの手動ランと所感。
- 強化間隔、最終強化、危険イベントの採否記録。

完了条件:

- 型確認、単体テスト、本番ビルド、主要E2E、画面比較が通る。
- 武器ごとに最低2本の手動ランを記録する。
- 未解決P0がなく、採用、調整、削除とv0.7への送りを説明できる。

### v0.6完了条件

- 各武器ランの違いを一文で説明できる。
- 5分ランで最低1つの挙動変化型強化が機能する。
- 90秒以降も選択または予兆付き戦闘展開の節目が続く。
- PulseとSpreadに致命的な死に選択がなく、Pierceを通常選択へ出すか判断済みである。

## v1.0 最初のローカル完成版

v1.0はバックエンドを含むサービス公開ではなく、ローカルで完結する最初の完成版です。

完了条件:

- 初回ランからリザルトまで迷わず進める。
- 30分のプレイで複数のビルドまたはモードを試す理由がある。
- 標準ルールセットの自己記録を信頼して比較できる。
- 保存移行、初期化、設定が壊れない。
- 重要E2E、画面比較、シミュレーション、バランス比較が通る。
- 既知制約と次バージョン候補が文書へ残る。

## 根拠が出るまで延期するもの

- 外部ログインとログアウト。
- クラウド保存。
- オンラインランキングと不正対策。
- 日替わりチャレンジのバックエンド。
- 外観販売、課金、広告。
- 完全自動生成ステージ。
- 大量コンテンツ制作。

アカウント機能は、複数端末で続けたい需要またはオンラインチャレンジの価値を確認できた時点で再評価します。

## 実装開始時の順序

1. v0.4の操作未決事項を短い判断チケットで閉じる。
2. `PH-V05-001` でラン記録、ルールセット版、ランキング対象判定を定義する。
3. `PH-V05-002` と `PH-V05-003` で実際の手動ランをリザルトと履歴へ接続する。
4. v0.5終了時に画面改善の残量を判断する。
5. `PH-V06-010` で弾、貫通、無敵時間ルールを固定する。
6. `PH-V06-008` で変更前の計測契約を定義する。
7. `PH-V06-009` でv0.6が変更する画面とデバッグ責務を分ける。
8. `PH-V06-007` で用途別乱数列を分離する。
9. `PH-V06-001` で成長間隔をシミュレーション比較する。
10. `PH-V06-002` でPulseとSpreadをラン開始前選択へ出す。

この時点までは、新ステージやボスの本実装へ入りません。
