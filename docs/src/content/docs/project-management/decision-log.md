## 2026-06-29: v0.3安定版候補

決定: v0.3を安定版候補として扱う。

理由:

- 最終検証が通った。
- 回復ピックアップはゲームを壊していない。
- 画面外敵方向表示は維持してよい。
- 回復量の追加調整は不要と判断した。
- 残リスクはv0.4以降へ送る。

## 2026-06-29: 最初のアイテム候補

決定: 最初のアイテム試作候補を `haste` とする。

理由:

- 開始、継続、終了が明確である。
- 統計とラン出力で確認しやすい。
- 攻撃力を直接上げない。
- 回復と役割が重なりにくい。

## 2026-06-29: 長期やりこみの順序

当初決定: v0.4で操作性、v0.5でステージ、v0.6で装備、v0.7でラン外進行を扱う。

現在の状態:

- 2026-07-08にv0.5をエンドレスの磨き込みへ変更した。
- 2026-07-10にビルドの個性をv0.6、戦闘展開とステージ試作をv0.7へ置いた。

当初の理由:

- 操作感が不安定なまま装備やラン外進行を入れると、調整原因が分からなくなる。
- ステージは障害物、ウェーブ、敵構成を束ねるため、装備より先に土台化する価値がある。
- 装備と解放は保存が絡むため、最初から大量実装しない。

## 2026-06-29: v0.4障害物の初回対応

決定: 中央障害物を削除し、跳弾は通常武器へまだ付与しない。

理由:

- 中央ブロックは開始位置付近の移動と射線を塞ぎやすい。
- 通常武器へ跳弾を付けると、バランス回帰テスト上の生存時間が大きく伸びた。
- 跳弾自体は有望なため、設定、データ、シミュレーション処理は残す。
- 後続で武器最終強化、ステージ特殊ルール、チャレンジとして評価する。

## 2026-06-29: 自動射撃試作

決定: プレイ中にマウス照準が確立されたら、Phaser入力アダプターで自動射撃する。

理由:

- `WASD + マウス照準 + 射撃維持` の同時操作負荷が高い。
- シミュレーションの入力境界を変えず、小さく試せる。
- Spaceキーを将来のダッシュまたは防御行動候補へ空けられる。

制約:

- メニュー操作中のマウス移動は照準として保持しない。
- 左クリックとSpaceキーの射撃は互換用として残す。
- バランス回帰テストは、人間にとっての入力改善を直接評価しない。

## 2026-07-01: v0.4から次フェーズへ進む判断

当初決定: 現在のバランスを即時変更せず、v0.4の操作方針を閉じた後にステージ基盤へ進む。

現在の状態:

- 2026-07-08にステージ基盤をv0.6へ移した。
- 2026-07-10に戦闘展開と合わせてv0.7へ移した。

理由:

- 手動プレイで約23000点のランが出たが、毎回再現した結果ではない。
- 従来の最高記録は約8000点で、通常は5000点未満が多かったため、突出値は上達余地として扱う。
- 最後は操作ミスで死亡しており、死因の納得感がある。
- 1回の高スコアだけで難度を変えると、操作慣れ、運、設定差を誤読しやすい。

制約:

- 高スコア、通常、短時間のランをラン出力で残す。
- 3本から5本の記録が揃うまでは、突出値だけで難度を変えない。
- 20000点以上が頻発し、緊張感が薄れる場合は再評価する。

## 2026-07-07: シードのランダム化と持久圧力

決定: ステージ基盤より先に、手動ランのシード変更と90秒以降の持久圧力を入れる。

理由:

- 自動ログで51282点、約882秒、レベル20の長時間ランを確認した。
- 強化後に長時間続けられそうな感触があった。
- ステージやボスを待たず、現在のアリーナでもラン差と終盤圧力を少し足す価値がある。
- 大幅な弱体化ではなく、シード差と小さな持久圧力に留める。

制約:

- 自動テストは固定シードを維持する。
- 手動ランの再現には `?seed=` を使う。
- 90秒以降の圧力には上限を持たせる。

## 2026-07-08: v0.5をエンドレスの磨き込みへ変更

決定: 次フェーズはステージ基盤へ直行せず、v0.5を「エンドレスの磨き込みとラン記録」とする。

理由:

- 現在のエンドレスが遊べる状態になり、リザルト、ランキング、履歴で振り返れる価値が高い。
- タイトル、HUD、演出、音を少し磨くと、次のプレイテストから得られる情報が増える。
- 外部ログインより先に、ローカルプロフィールとラン記録の境界を作る方がよい。
- ステージやボスへ進む前に、保存、表示、計測を固めると比較しやすい。

制約:

- 外部アカウント同期、オンラインランキング、バックエンドはv0.5で作らない。
- 無料BGMや効果音は、ライセンスと出典を確認してから取り込む。
- リザルトとランキングはゲームルール版ごとに分ける。

## 2026-07-10: ステージ追加よりビルドの個性を優先

決定: v0.5後はステージ基盤へ直行せず、v0.6を「ビルドの個性とエンドレス後半」、v0.7を「戦闘展開とステージ試作」とする。

理由:

- 51282点、約14.7分のランはレベル20、強化19回で、次のレベルに1792 XP必要だった。
- 全強化ランク24へ届くレベル25まで、現行式では累積約24088 XP必要になる。
- 開始武器が `pulse` 固定で、ラン開始時の方針が1つしかない。
- 90秒以降は敵構成の重みが同じまま、数値圧力だけが上がる。
- 同じアリーナでもビルド差が生まれる状態を先に作ると、後続ステージの価値を正しく評価できる。

制約:

- v0.5のラン記録、リザルト、ランキング基盤は予定通り先に閉じる。
- v0.6は開始武器、選択間隔、最初の武器最終強化へ範囲を絞る。
- ステージ、エリート、ボスは戦闘展開制御と出現安全規則の後に試す。
- アカウント同期、オンラインランキング、恒久ステータス周回はv1.0前の必須範囲にしない。
- ゲームルール変更時は `rulesetVersion` を分け、デバッグと自動テストを通常ランキングへ混ぜない。

## 2026-07-10: v0.4の終了と操作案の延期

決定: 自動射撃をv0.5の既定操作として暫定採用し、防御ダッシュ、右クリック固有スキル、跳弾の実戦投入はv0.6以降へ延期する。

理由:

- 自動射撃を含む現在の操作で、短時間から長時間まで複数の手動ランを完走できた。
- メニュー、強化、リザルトのカーソル問題は修正済みである。
- 防御行動と固有スキルを追加すると、v0.5の保存、画面、設定とは別のゲームプレイ調整が必要になる。
- 跳弾は通常性能へ入れると生存時間が大きく伸びる可能性があり、武器最終強化と合わせた方が役割を判断しやすい。

制約:

- 左クリックとSpaceはv0.5の手動射撃として残す。
- 自動射撃は `PH-V05-010` で切り替え可能にする。
- 延期項目は削除せず、v0.6以降のゲームプレイ試作として再評価する。
- 延期項目をv0.5の未完了条件へ含めない。

## 2026-07-10: v0.5を11チケットへ再編

決定: 既存9チケットに、設定とアクセシビリティ境界、安定化と手動プレイ判定を追加し、v0.5を11チケットで管理する。

理由:

- 音、揺れ、点滅、自動射撃の設定が複数機能へまたがるため、音響やプロフィールの付随作業にすると所有者が曖昧になる。
- 個別機能の完了だけでは、履歴の永続化、ランキング除外、長時間動作をまとめて判定できない。
- 開発用ランJSONは存在するが、プレイヤー記録、保存契約、プロフィールは未実装であり、実装済みと誤認しやすい。
- GitHub Issueが0件だったため、設計資料と実行管理の役割を明確に分ける必要があった。

運用:

- Starlightの [v0.5チケット詳細](../v05-tickets/) を要件の正本とする。
- [GitHubマイルストーン](https://github.com/garchomp-game/create-game/milestone/1) を担当、進捗、実装中の議論の正本とする。
- `PH-V05-001`を最初の実装チケット、`PH-V05-011`をフェーズ全体の最終判定とする。

## 2026-07-10: v0.5リリース候補とルールセット維持

決定: `PH-V05-001`〜`010`の実装と`PH-V05-011`の自動検証を完了し、v0.5をリリース候補とする。アプリ版は0.5、ゲームルール版は`phaser-v0.4-endless-pressure`を維持する。

理由:

- 記録、履歴、ランキング、プロフィール、設定、タイトル、HUD、演出、音響を接続した。
- 保存の一回性、破損、容量超過、件数上限、再読込、個別初期化を自動確認できた。
- 18状態の画面比較と900秒耐久シミュレーションが通った。
- 実装上の未解決P0は見つかっていない。

制約:

- 旧v0.4ログは`rulesetVersion`を持たないため、v0.5ランキング比較データへ移行しない。
- 900秒試験は加速シミュレーションであり、実時間GPU負荷と人間の操作感を代替しない。
- 15.1分の実時間ブラウザ耐久は完了した。v0.5履歴へ短時間、通常、長時間の3ランを残し、画面・音響の所感を確認してから`PH-V05-011`を閉じる。
- v0.6の設計準備は並行できるが、ゲームルール変更は手動ゲート後に始める。
- Scene / Rendererの追加分割とバンドル警告は、v0.6開始時に変更範囲と合わせて再評価する。

## 2026-07-10: v0.5音源はプロジェクト生成素材を使う

決定: 外部の無料素材を急いで採用せず、FFmpegで本プロジェクト向けに生成したBGM1曲と主要効果音8種をv0.5の基準素材とする。

理由:

- BGM制御、効果音ルーティング、音量、ミュート、ブラウザ音声ロックの基盤確認が主目的である。
- 外部素材のライセンス、再配布、クレジット条件を持ち込まずに検証できる。
- 外部素材へ差し替える場合も、`public/audio/README.md`を資産台帳として使える。

## 2026-07-11: v0.5完了と音響の再構成

決定: 通常UIからの手動3ラン、15.1分耐久、全自動検証、所感反映が揃ったため`PH-V05-011`を完了し、v0.5を閉じる。音響は外部曲へ急いで差し替えず、再生成可能な32秒BGMとSEバリエーションをv0.5採用品とする。

根拠:

- 約98秒、178秒、272秒の3ランが、同じv0.5ビルドのブラウザ履歴、ランキング、開発ログへ保存された。
- 3ランはすべて`manual`起点、ランダムシード、ランキング対象だった。
- 操作とUIは良好と評価され、初版SEの単調さが唯一の具体的な改善指摘だった。
- 射撃、命中、撃破、回収、被弾へ2〜3バリエーションを追加し、BGMを8秒から4区間・32秒へ再構成した。
- 18ファイル・121件の単体テスト、43件の通常E2E、18状態の画面比較、本番ビルドが再度成功した。

後続判断:

- 外部無料BGMは、ゲームプレイ拡張より優先しない。
- 採用候補が見つかった場合は、選曲比較とライセンス台帳更新を別チケットで行う。
- Scene / Renderer分割、バンドル警告、開始武器、XP曲線はv0.6へ送る。
