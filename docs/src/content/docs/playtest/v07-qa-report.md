## 判定

`0.7.0` / `phaser-v0.7.0-final-expedition-rc3`を**第10最終遠征の手動再評価用preview候補**とします。既存v0.6.8 productionは変更しません。

自動ゲートでは、両武器が最終段階へ到達し、各武器に少なくとも1勝があり、不可避攻撃と上限逸脱がないことを確認しました。全勝を要求しない高難度最終面として到達性は成立しています。

production昇格はPulse / Spread各3本の通常UIランまで保留します。自動入力は到達不能、再現性、密度上限を確認できますが、面白さ、納得感、人間の技能負荷は代替しません。

## 可避性と説明可能性

- 両武器で固定照準斉射を無被弾回避するfixtureが通る。
- ボス追跡は既存の障害物経路を使い、障害物を貫通しない。
- 挟撃増援は安全距離、活動領域、障害物、到達可能性、敵上限を共有する。
- 斉射と護衛抑制射撃は攻撃IDへ帰属し、最後の攻撃種をRun JSONへ残す。
- 通常ウェーブ、護衛、ボス、敵弾は共通上限を使う。
- ボス撃破による勝利、保存、`game.over`は一度だけ発生する。
- desktopとportraitでCommander 3行バナー、HP 500、ボスHP 3400、広域予告に重なりがない。

## 既知事項

- ceiling観戦AIではSpreadが3シードとも安定勝利し、Pulseは長期戦になった。人間の武器採否へ直接使わない。
- 最終ボスは道中より明確に高密度になった。第1から第9の初心者向け難度へ流用しない。
- game JSはViteの500KB chunk警告を超える。gzip 474.25KBで、今回の変更による停止条件にはしない。
- `esbuild 0.27.7`相当のLow 1件はWindows開発サーバーの任意ファイル読み取りで、Linux上の静的公開成果物へ直接影響しない。
- 世界観、最終アート、ステージ別ボスBGM、第1から第9、進行保存は未実装である。

## 手動採否

RC1 / RC2の手動ランは調整根拠として残しますが、RC3のproduction採否本数には混ぜません。更新Previewで各武器3本を、デバッグhookと観戦AIを使わずに行います。

記録する項目:

- ボス到達、勝敗、終了時刻、スコア、到達Act、ボス残HP。
- 大型黄が単なるボーナスではなく、狙いやすい耐久目標として機能したか。
- 黄緑の初出が難度段差として読み取れ、理不尽な出現になっていないか。
- Commander HP 500が長すぎる停滞か、意味のある優先標的か。
- ボス追跡、通常増援、広域斉射に対して射線離脱か障害物利用で対応できたか。
- Pulseの優先標的・射線維持とSpreadの群れ処理が別の強みとして感じられたか。
- 再挑戦したい理由が、攻撃理解、武器、操作改善のいずれかにあったか。

重大 / 高の再現不具合が0件で、両武器がボスへ到達し、各武器で少なくとも1勝できればproduction昇格候補とします。高難度最終面なので、各3本の全勝は要求しません。
