## 目的

プレイヤーが今の状態、選べる行動、被弾理由、ラン結果を短時間で理解できる画面を作ります。

装飾量より情報の優先順位を先に整えます。

## 現在の画面評価

2026-07-10時点の画面画像で確認した初期課題です。v0.5からv0.7でカーソル、高解像度選択、可変桁HUD、結果要約は改善しましたが、タイトルの個性、情報階層、共通theme tokenは次段に残ります。

- タイトルは中央の説明と開始ボタンだけで、モードやゲームの個性が見えにくい。
- `ライブラリ: Phaser` はプレイヤー向け情報ではなく、製品画面から外す必要がある。
- HUDはHP、XP、ウェーブ、時間、スコア、敵数、武器性能が左上の同じ階層へ集まっている。
- 強化画面は数値差分を表示できているが、役割、相乗効果、最終強化への進行が区別できない。
- RC5のリザルトは3列へ整理し、表示文をPhaser非依存のResult Presenterへ分離した。Canvas内の固定座標はPassive Viewへ残している。
- タイトル、選択、リザルト、HUDの現行色は共有theme tokenへ移した。次は比較草案で文字階層と余白を評価する。

## タイトルとラン開始前

- 最初の画面からエンドレスを1回から2回の操作で開始できる。
- 単なる「開始」ではなく、選ぶモード名を表示する。
- ランキング、履歴、設定は主開始ボタンと競合させない。
- 未実装モードを大量に並べない。
- ステージと武器が増えたら、ラン開始前設定画面へ段階的に移す。
- 背景演出を入れる場合は、実際のゲーム描画を使った軽いデモを候補とし、開始を遅らせない。

## プレイ中HUD

情報を安定した領域へ分けます。

| 領域 | 情報 |
| --- | --- |
| 左上 | HP、XP、レベル |
| 右上 | スコア、時間、危険度 |
| 画面下 | 防御行動、固有スキル、再使用時間 |
| ボス戦上部 | ボスHP、段階変化 |

要件:

- 武器の詳細数値を常時文章で並べず、アイコンと一時停止画面の要約へ寄せる。
- 危険度警告、ボス表示、強化画面を重ねない。
- 可変内容がHUDの大きさを変えないよう、表示領域を固定する。
- 色だけで準備完了や危険を伝えない。

## 強化選択

- 現在ランクと取得後ランクを表示する。
- 数値は「現在 -> 取得後」で示す。
- 武器、機動、生存、補助、スキルの分類を表示する。
- 最終強化の条件と進行を短く示す。
- 最大ランク、取得不可、条件不足を区別する。
- キーボードとポインターのどちらでも選択できる。
- PCでは選択画面を開いた直後から`1`、`2`、`3`で対応候補を決定できる。Tabによる事前focusを要求しない。
- デスクトップでは3候補を横比較し、狭い画面では縦並びへ切り替える。
- 候補数が減っても画面全体の配置をずらさない。
- 通常強化、EX強化、後半契約では戦場を停止したまま透過表示し、再開時の敵、弾、障害物、退路を選択前に確認できる。
- 選択後は直前のpointer照準と押下中の移動入力を引き継ぐ。無敵時間は追加せず、入力を失ったことによる被弾だけを防ぐ。

## リザルト

最初の画面で次を確認できるようにします。

- スコアと自己最高記録との差。
- 生存時間、レベル、撃破数。
- モード、ステージ、開始武器。
- 主な強化と最終強化。
- 最後のダメージ源。
- シードとルールセット。
- ランキング対象かどうか、その理由。

RC5の960 x 540では、左にラン概要、中央に操作、右にビルドと詳細を置く3列へ分けました。Expedition勝利は完遂、速攻、合計、ボス戦時間を同じ階層で比較できます。詳細履歴は別画面に置き、枠の中へさらに枠を重ねず、見出しと余白で区切ります。

致死時の赤い画面効果は短時間で収束させ、リザルトの背景色と文字コントラストを支配しないようにします。

## 履歴とランキング

- ラン履歴は直近50件を上限候補とする。
- ランキングは比較条件ごとに上位10件を保持する。
- 標準外シード、デバッグ操作、テスト状態は通常ランキングと分ける。
- 設定変更やルール更新で比較不能になった記録を混ぜない。
- 記録削除は履歴、プロフィール、設定を個別に扱えるようにする。

## 設定とアクセシビリティ

- BGM音量、効果音量、ミュート。
- 画面揺れの強さ。
- 画面点滅の強さ。
- 自動射撃の切り替え候補。
- 色と形を併用した敵識別。
- キーボードとポインターの両対応。

## 演出と音

優先するフィードバック:

- ダメージを与えた、敵を倒した。
- 回復した、回復できなかった。
- レベルが上がった、最終強化を取得した。
- 防御行動が使用可能になった。
- 自己最高記録を更新した。
- ボス攻撃と段階変化が始まった。

無料素材を使う場合は、出典、ライセンス、加工可否を資産一覧へ残します。BGM制御は効果音処理から分け、ループ、音量、ミュート、画面遷移を独立して扱います。

## 表示実装の方針

- HP、XP、スコア、ボス状態はdomain / simulationが所有し、見た目の都合で戦闘状態を増やさない。
- 画面文、表示順、要約は`ArenaResultPresenter`、`ArenaSecondaryMenuPresenter`、`ArenaChoicePresenter`などのPhaser非依存Presenterが作る。
- 戦闘HUD、予告、弾、障害物はworldとの重なり順が必要なためWebGLへ残す。
- 武器、強化、契約、将来のステージ選択やフォームはsemantic DOMを使い、高解像度文字、focus、pointerを維持する。
- `ArenaTheme`を正本としてPhaser数値色とCSS custom propertiesを派生させる。敵、弾、演出色は視覚案の採用まで別管理とする。
- React / Vue / Svelteや重量級component libraryは先行導入しない。Tailwindはproduction外の比較prototype、Lucideは既知操作のDOMアイコン候補として採否する。

比較prototypeでは、戦術管制、回収航路、精密アーケードの3案を同一データとviewportで検証しました。本番は戦術管制の情報階層を基礎とし、回収航路は遠征進行、精密アーケードは武器成果とリザルトへ限定して使います。最初のproduction縦切りとして、責務分離済みの武器・通常強化・EX強化・契約選択画面へ反映しました。

選択画面は画面文脈、候補番号、役割、ランク、説明、取得後数値、選択コマンドを別領域へ置きます。武器・機動・生存・補助・最終・EX・契約は、色だけでなくラベルとmarker形状を併用します。desktopでは候補を下段へ寄せて停止中の戦場を残し、portraitでは固定行へ積みます。全候補の文字とfocus領域がviewport内へ収まること、`1`から`3`の即時選択、選択後の照準継続をPlaywrightで検査します。

具体的な責務分割と実装順は[UI・グラフィック再設計計画](../../project-management/ui-visual-redesign-plan/)、採否と再検討条件は[UIライブラリと視覚方向](../../engineering/ui-library-and-visual-direction-adr/)を正本とします。

## v0.5の完了条件

状態: 2026-07-10に実装と画面比較を完了。

- エンドレスを2操作以内で開始できる。
- 製品画面にライブラリ名やデバッグ表示が残らない。
- HP/XP、スコア/時間、スキルの情報領域が分かれる。
- リザルトでスコア、死因、ビルド、シード、自己記録との差を確認できる。
- 致死時の画面効果が結果情報を覆わない。
- 音、揺れ、点滅の設定を保存できる。

実装順は [v0.5作業計画](../../project-management/endless-polish-plan/)、画面ごとの受け入れ条件は [v0.5チケット詳細](../../project-management/v05-tickets/) を参照してください。
