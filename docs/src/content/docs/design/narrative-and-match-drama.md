## 議論から見えた中心

現在のArena Coreは、操作の軽さ、混戦時の見やすさ、照準と位置取りが成果へ反映される点に強みがあります。グラフィックや物語を追加するときも、この強みを装飾で覆わないことを前提とします。

目指す体験の草案:

> 避ける、狙う、欲張る。上達が生存時間を伸ばし、同じ生存時間では武器固有技術とリスク選択がスコア差になる、1画面エンドレス競技アリーナ。

「長く生き残るほど高得点」を主軸に保ちます。精密射撃、複数標的処理、危険イベント攻略などの技量は、副次的な差としてスコアと演出へ返します。

## 物語を追加する段階

物語性は、最初から長編シナリオや会話イベントを意味しません。次の3段階を分けます。

| 段階 | 役割 | 優先度 |
| --- | --- | --- |
| 世界の前提 | なぜ戦い、敵やXPが何を表すかを決める | 高 |
| 1ランのドラマ | 危機、選択、逆転、最終局面を作る | 高 |
| ランをまたぐ物語 | 登場人物、記録、ステージ間の展開を作る | v1以降 |

公開ベータからv1手前では、世界の前提と1ランのドラマを優先します。プレイ中に長い会話やカットシーンを挟みません。

## 推奨候補: 艦隊防衛シミュレーション

現在の抽象的な図形、ランキング、固定シード、繰り返し挑戦を無理なく説明できる候補です。

> 戦術シミュレータが再現する最後の艦隊防衛戦。プレイヤーは機動兵器COREを操作し、防衛線が崩壊するまで敵艦隊を食い止める。

ホログラフィックな戦術表示とすれば、現在の幾何学表現を仮素材ではなく意図したスタイルへ発展させられます。戦闘が現実の作戦なのか、過去の敗戦記録なのか、未来の操縦士を選ぶ試験なのかは後から決められます。

| 現在の要素 | 世界観上の候補 |
| --- | --- |
| プレイヤー | 機動兵器CORE、迎撃艇、遠隔戦闘端末 |
| Chaser | 標準迎撃艇 |
| Fast | 高速突撃艇 |
| Brute | 重装巡洋艦 |
| Ranged | 砲撃艦 |
| Pulse | 高速精密砲 |
| Spread | 迎撃用プラズマ散弾 |
| 障害物 | 防衛設備、シールド発生器、残骸 |
| XP | 回収したエネルギーまたは戦闘データ |
| 回復 | 修復セル |
| EX強化 | COREの出力限界解除 |
| アリーナ崩壊 | 防衛シールドの段階的消失 |
| シード | 再現する戦況ID |
| ランキング | 操縦評価、戦果記録 |

### 他候補との比較

| 候補 | 現行表現との相性 | 素材負荷 | 物語の広げやすさ | 主な懸念 |
| --- | --- | ---: | ---: | --- |
| 艦隊防衛シミュレーション | 高い | 中 | 高い | 抽象表現だけでは感情的な距離が残る |
| 実在する宇宙戦の最終防衛 | 高い | 中 | 高い | エンドレスと完全勝利を両立しにくい |
| 魔物による都市襲撃 | 中 | 高 | 高い | 有機的な敵、地形、アニメーションの負荷が大きい |
| 純粋な競技シミュレータ | 非常に高い | 低 | 中 | 戦う理由と危機感が弱い |

艦隊防衛とシミュレータを組み合わせる案を第一候補とし、視覚試作を見て採否を決めます。

## エンドレスと勝利条件

敵艦隊を完全に倒す物語は、最終的に安全領域が消える現行エンドレスと衝突します。モードごとの目的を分けます。

| モード候補 | 物語上の目的 | 終了 |
| --- | --- | --- |
| Last Stand | 避難、解析、増援到着まで戦果と時間を稼ぐ | 防衛線崩壊または撃墜 |
| Counterattack | 艦隊を突破し、敵旗艦を倒す | ボス撃破による勝利 |
| Challenge | 指定装備や特殊戦況で評価を競う | 条件達成または撃墜 |

現行エンドレスはLast Standとして扱えます。将来のステージ・ボスモードはCounterattackとして明確な勝利を持たせます。

## 1ランのドラマ

MOBAのような1試合の起伏は、物語テキストより、育成、目標、危機、パワースパイク、立て直し、最終決戦から生まれます。Arena Coreには時間帯、危険イベント、EX進化、ビルド完成、崩壊という土台があります。

| 時間帯 | 現在の仕組み | ドラマ上の役割 | 表現候補 |
| --- | --- | --- | --- |
| 0から30秒 | 基本敵と最初の強化 | 初接敵と準備 | 静かなグリッド、索敵表示 |
| 30秒以降 | 高速・重量敵の追加 | 最初の圧力上昇 | 防衛灯、遠方の艦影 |
| 約2分20秒 | 最初の危険イベント | 最初の大規模襲撃 | 指揮艦、警報、進入方向の予告 |
| 4分前後 | CORE完成とEX選択 | ランの方針を決める転換点 | Protocol選択、機体と弾の段階変化 |
| 4から6分 | EX進化と脅威上昇 | パワースパイクと敵側の追随 | Evolution、複合編成、強まる警報 |
| 6から10分 | 脅威上昇と反復イベント | 敵味方双方のエスカレーション | 複合艦隊、局所目標、損傷した防衛設備 |
| 10分以降 | アリーナ崩壊 | 最終防衛戦 | シールド消失、旗艦級脅威 |

毎ランを「初接敵、最初の危機、戦術選択、ビルド完成、反撃、防衛線崩壊」という流れで読める状態を目標にします。

## ドラマを作るゲーム要素

単純な敵数増加だけではなく、局面ごとに固有の主体、目標、結果を置きます。

- 危険イベントへ指揮艦または中ボスを加える。
- 指揮艦を先に倒すと増援や砲撃が弱まり、回収時間を得られる。
- 砲撃座標、補給地点、防衛装置など、一時的に移動を要求する目標を出す。
- 一度撤退した敵が強化されて再登場し、ラン内の因縁を作る。
- イベント突破後に短い回収・修復区間を置く。
- EX Protocolの選択を、以降の攻撃表現、成長の手触り、戦術判断へ継続して反映する。
- 敵の攻撃を別の敵へ当てるなど、危険イベントを上手く使う余地を作る。

一時目標は失敗した瞬間にランを終了させず、その後の圧力や報酬を変える形から試します。

## 視覚表現の原則

- 背景の変化は現在の危険度と戦況を表す。
- スコア、自己記録、熟練表示はプレイヤーの成果を表す。
- 敵の基本色と輪郭は役割識別のため維持する。
- 耐久上昇は一律の色替えではなく、装甲リング、発光コア、外装段階で示す。
- 床テクスチャは低コントラストとし、弾、敵弾、XPより目立たせない。
- 全画面Bloomや常時画面揺れへ依存せず、設定した点滅・揺れ強度を守る。
- 危険イベント、通常ビルド完成、EX移行、崩壊を、背景、境界、音の節目として揃える。

## 技量とスコア

基礎スコアは敵撃破点と脅威上昇を維持し、長時間生存を最も大きな得点要因とします。技量ボーナスを導入する場合は、同程度の生存時間を分ける副得点として始めます。

初期仮説:

- 技量ボーナスは総得点の10%から20%程度を目安にする。
- Pulseは精密コア、最大集束撃破、複数貫通、反射追撃を評価する。
- Spreadは複数標的斉射、掃射連鎖、近距離の扇状制圧を評価する。
- 共通評価は危険イベント無被弾突破、優先標的撃破、危険距離での撃破を候補にする。
- 危険イベント中だけ倍率を上げ、上手く利用した時間をボーナスタイムにできる。

中心命中は、現在の敵マークを可視コアへ発展させる案と相性がよい一方、Pulseには最大+80%の集束報酬が既にあります。最初の試作では追加ダメージを与えず、スコア、演出、計測だけで価値を確認します。観戦AIは中心を高精度に狙えるため、人間向け評価とAI比較を同じ結果へ混ぜません。

## XP回収

完全自動回収は、危険な場所へ近づくかという位置判断を消します。標準ルールでは現在の磁力回収を維持し、回収の気持ちよさと後処理を改善します。

- 密集XPを大きな結晶へ統合する。
- 危険イベント終了後だけ短時間の回収波を発生させる。
- 磁力範囲を強化、装備、ステージ効果として扱う。
- 期限付きXPは通常ドロップではなく、特殊イベント用の高価値資源として試す。

## 物語の伝え方

- 開始前の短い作戦名と目的。
- 危険イベント開始時の短い通信またはコールサイン。
- 指揮艦、シールド損傷、増援を背景と登場演出で示す。
- リザルトを戦果報告として構成する。
- 任意で読める戦闘記録をラン外に解放する。

プレイを止める長い説明、毎ラン同じ会話、スキップ前提の導入は避けます。

## 最初の縦切り

2026-07-17の実装比較では、次の最小表現を採用しました。

- 低コントラストのグリッド、同心円、座標軸を持つ戦術表示背景。
- 暗い六角船体、明るいコア、左右の張り出しで通常敵から区別するCommander。
- 外周の帯と山形マーカーで示す侵入方向、拡張する枠とリングで示すAct遷移。
- Chargerの後部フィンと予告線を組み合わせた進行方向の補助表示。

弾、敵弾、XP、回復の既存色を優先し、背景は装飾ではなく座標と戦況を読む補助に留めます。この方向は採用しますが、具体的な艦船テクスチャ、人物、固有名詞までは固定しません。次のボス縦切りでも同じ輪郭、コア、戦況色の文法を使い、画面密度を増やす前に可読性を再評価します。

1. 世界設定を1ページに固定する。
2. 0分から10分以降までの試合展開を時系列で固定する。
3. 現在の敵4種を艦艇として再解釈した視覚案を作る。
4. 背景1種、敵1種、危険イベント1種だけを試作する。
5. 指揮艦を使った中ボス戦を1件試作する。
6. 最大密度の視認性とWebGL性能を確認してから素材量を増やす。

グラフィック実装は、[Phaserプレゼンテーション再設計](../../engineering/phaser-presentation-architecture/)の責務分離後に進めます。

## 現行assetの事前監査

2026-07-20時点では、テーマを選ぶ前に置換必須となる外部assetはありません。

| 領域 | 現在の実装 | #66での扱い |
| --- | --- | --- |
| 背景 | `PhaserTacticalBackground`が低コントラストの格子、同心円、座標軸、境界を一度だけ描画 | 座標を読む機能は維持し、stage差はtokenと小さいlayer追加で比較する |
| 敵 | `Phaser.Graphics`の円、正方形、菱形、六角形と役割mark | 色を変える前に輪郭、突起、装甲、砲口の文法を足す |
| Commander / Boss | 多角形の船体、左右wing、発光core、回転ring | 既存の識別性を基準にし、bitmap化を採用条件にしない |
| 警告 | 黄の予告、橙・coralのactive、線、帯、ring | 背景themeより常に高い優先順位を維持する |
| UI font | OS内の`Noto Sans CJK JP`等へfallbackするlocal stack | 外部fontを先に同梱せず、必要文字、可読性、配布条件を比較してから判断する |
| BGM / SE | リポジトリ内スクリプトで決定的に波形生成した18ファイル | 外部録音なしのbaselineを残し、stage別音楽は別candidateとして比較する |
| bitmap / web font | productionには未導入 | 出典と再配布条件を確認できない素材は候補へ入れない |

現在はCanvas上のベクター描画なので、拡大率ごとの低解像度spriteを抱えていません。一方、色値は`ArenaWorldView`、HUD、feedback、CSSへ分散しています。theme採用前に共通tokenへ一括移動せず、採用後に実際の変更理由が重なる背景、警告、敵外観から段階的に抽出します。

## 比較する3つの物語フレーム

3案はレイアウトや戦闘ルールを変えず、同じRC6画面の固有名詞、背景差、開始文、勝敗文だけで比較します。

| 案 | 一文の前提 | 四方侵入 | Endless / Expedition | 強み | 主な懸念 |
| --- | --- | --- | --- | --- | --- |
| A 戦況再演記録 | 失われた防衛戦を戦術記録から再演し、より良い迎撃手順を探す | 記録上の包囲座標として自然 | 生存記録と旗艦撃破を別simulationとして扱える | fixed seed、再挑戦、抽象図形、ランキングと最も整合 | 現実の危機より感情的な距離がある |
| B 避難回廊防衛 | 包囲された中継核の周囲で避難航路を維持し、最後に敵指揮中枢へ反撃する | 防衛周縁の破口から侵入する | Endlessは時間稼ぎ、Expeditionは反撃作戦 | 守る理由、回復、障害物、stage進行を説明しやすい | 艦隊や施設を具体化するとasset負荷が上がる |
| C CORE適性試験 | 操縦者が再現戦況へ挑み、武器ごとの戦術適性を証明する | 試験arenaの投入方向として自然 | どちらも評価課題として一貫する | スコア、PB、技能ledger、教義と強くつながる | ボスや撤退の物語上の切迫感が弱い |

比較の第一候補は、**Bの守る理由をAの戦術再演表示で伝える案**です。実戦そのものを全面描写せず、「避難回廊が失われた戦況記録を再演し、Stage 10で指揮中枢を突破する」とすれば、現在の抽象表示と再挑戦を維持しながら目的を具体化できます。

これは採用決定ではありません。#83で緊張・緩和、再挑戦、Stage 1 / 5 / 10の役割を固定した後、A単独、B単独、第一候補の3画面を同じfixtureで比較します。Cは技能ledgerとランキングの表示語彙として残せますが、世界全体を試験設定へ固定する案とは分けます。

## 敵の視覚文法

現行の色と基本shapeを壊さず、シルエットだけで役割を推測できる方向へ伸ばします。

| 役割 | 現行shape | 追加する非色表現 | 動きとの対応 |
| --- | --- | --- | --- |
| 追跡体 | 円 | 進行方向の短い切欠き、単一core | 素直に接近する基準敵 |
| 重装体 | 正方形 | 二重装甲枠、鈍い四隅、広い被弾面 | 遅く、硬く、進路を塞ぐ |
| 高速体 | 菱形 | 後方fin、細い前端、短い残像 | 小さく速く横切る |
| 射撃体 | 六角形 | 砲口または照準ring、発射前の収束線 | 距離を保ち射線を作る |
| Commander | wing付き多角形 | 増援方向と同期する外ring | 先に倒すべき局所目標 |
| Charger | 基本shape + action | 進路線、後部fin、prepare / chargeの姿勢差 | 予告を読んで進路を外す |
| 最終指揮中枢 | 大型wing、core、ring | phaseで外装が開き、攻撃ごとに別の予告形 | 遮蔽、距離、標的優先を切り替える |

HP上昇だけを色替えで示しません。通常強化個体を追加する場合は装甲枠の枚数、core周囲のsegment、外装の開閉で段階を示します。

## Stage 1 / 5 / 10の画面仕様候補

| Stage | 背景 | 侵入・警告 | 音 | 画面で伝えること |
| --- | --- | --- | --- | --- |
| 1 基礎迎撃 | 境界が完全な格子、少ない損傷mark、低い背景密度 | 1方向ずつ大きく予告 | 薄いpulseと少ない打音 | 移動、照準、回収、最初の成功 |
| 5 四方包囲 | 一部区画の損傷、航路線、複数の外周segment | 優先方向を選べる順次警告 | 危険前だけ打音層を追加 | 退路を読み直し、回収と迎撃を両立する |
| 10 最終遠征 | 中枢座標、破損した境界、指揮中枢用ring | Commander、Charger、boss攻撃で形を分ける | Act、boss phase、勝利を別cueで区切る | 既習技能を切り替え、最終目標を撃破する |

stage差を画面全体の単色filterで表しません。背景の線、境界segment、損傷mark、音の層を変え、プレイヤー、敵弾、回復、危険色は全stageで同じ意味を維持します。

## #66の比較手順

1. 同じRC6最大密度fixtureから、通常終盤、危険予告、boss phase 2、portraitを固定する。
2. 3案で変えるのは作戦名、目的文、背景差、敵外装の語彙だけにする。
3. 敵種、危険方向、回復、プレイヤー位置を説明なしで識別できるか確認する。
4. 非背景pixel、描画p95、50ms超過、DOM overflowをbaselineと比較する。
5. 外部assetを使う案は、出典、作者、ライセンス、加工、再配布、クレジットを採用前に記録する。
6. #83のカーブと矛盾しない1案を採用し、不採用案の固有名詞とassetをproductionへ含めない。

採用までは新しいruntime library、web font、全敵sprite、stage別BGMを追加しません。

## 未決事項

- 戦闘は現実、過去の記録、未来予測のどれか。
- COREは有人機、無人機、戦術AIのどれか。
- Last Standで守っている対象は避難艦隊、惑星、防衛データのどれか。
- 語り手や指揮官を画面へ出すか。
- 技量ボーナスをランキング本体へ入れるか、熟練評価だけに使うか。
- 背景と敵をどこまで具体的な艦船へ寄せるか。

これらは文章だけで決めず、同じゲーム画面を使った2から3種類の視覚試作で比較します。
