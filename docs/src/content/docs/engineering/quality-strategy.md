## 基本コマンド

```bash
cd phaser
npm run typecheck
npm test -- --run
npm run test:e2e
npm run build
```

実時間15分のブラウザ耐久試験は、バランスによる死亡を防ぐデバッグ保護付きの描画・メモリ試験として明示的に実行します。

```bash
cd phaser
ARENA_LONG_SOAK=1 ARENA_HARDWARE_SOAK=1 npx playwright test tests/e2e/arena-soak.spec.ts --workers=1
```

このコマンドは計測hook付きの最適化済みビルドを生成し、`vite preview`とheaded Chromeを起動します。表示された試験Chromeは自動終了まで操作せず、他のゲームやGPU負荷を重ねません。開始時にWebGL rendererを記録し、SwiftShaderなら性能試験として扱いません。通常のE2Eではこの長時間試験をスキップします。
ゲームが有限終了することは、別の加速シミュレーションで検証します。
長時間モードでは計測負荷と巨大な失敗成果物を避けるため動画・traceを記録せず、JSON要約と失敗スクリーンショットを残します。通常E2Eは失敗時の動画・traceを維持します。

## 単体テストとシミュレーションテスト

`simulation`、`math`、`format`、設定スキーマはPhaserなしで検証します。

優先する観点:

- 乱数を固定すると同じ結果になる。
- 強化、ピックアップ、戦闘、移動を決定論的に確認できる。
- ゲームプレイ変更に合わせて統計値が壊れない。
- 用途別乱数列が互いに影響しない。
- 保存データの並び替え、件数制限、移行を純粋関数で確認できる。
- EX候補が同一サイクル内で重複せず、最後の1件が自動取得され、無効果候補を出さない。
- 敵の直線追跡、迂回、経路場共有、遮蔽物越しの射撃抑止が決定論的に動く。
- 危険イベントデッキが一巡するまで重複せず、収縮によって長時間試験が有限終了する。
- Pulseの集束段階とSpreadの掃射成立・消費が、武器をまたいで混入せず決定論的に動く。
- 役割別の敵HP成長は240秒までは基礎値を変えず、以降は設定した倍率順で増える。
- 最大Pulse弾速と50msステップでも敵をすり抜けず、貫通を交差時刻順に処理する。
- 同一フレームの往路命中、障害物または外周反射、復路命中と、Pulse専用機能の無効化を固定する。
- 反射後命中を障害物と外周へ帰属し、両者の合計が既存の反射後命中と一致する。
- 実フレーム時間の分位集計が外れ値、250ms超、リセットを正しく扱う。

## E2Eと画面比較

Playwrightで次を確認します。

- WebGLコンテキストが生成され、画面が空白にならない。
- タイトルからランを開始できる。
- 入力、一時停止、強化選択、ゲームオーバー、リスタートが動く。
- 固定状態の画面画像が意図せず変わらない。
- メニューとプレイ中でカーソルが正しく切り替わる。
- ラン記録と設定が再読み込み後も残る。
- EXサイクル、危険イベント予告、アリーナ崩壊が960 x 540と狭い表示で重ならない。
- 武器、通常 / EX強化、契約の選択肢が意味を持つDOMボタンとして操作できる。
- DPR 2でも選択UIがゲーム画面と一致し、390 x 844では一列に収まり、文字がラスタ拡大されない。

UIや視認性を変更した場合は、画面画像を更新または追加します。

## バランス回帰テスト

`balanceProbe` はAI入力モデルによる回帰検知です。

用途:

- 生存時間、毎分撃破数、毎分スコアの急変を見つける。
- 固定シードで変更前後を比較する。
- 敵数、弾数、回復量の上限逸脱を見つける。
- PulseとSpreadの生存、毎分撃破、毎分スコアを20%許容内で比較する。
- 命中ボレーごとの異なる敵数と固有能力の成立回数から、単線集中と広角制圧の差を確認する。

人間の操作感、楽しさ、被弾の納得感を判定するものではありません。手動プレイの代替にはしません。

基準値は [バランス回帰テスト](../../playtest/balance-probe/) にまとめます。

## ゲームプレイ試作の判定条件

新しい武器、強化、スキル、戦闘展開、ボスは、実装前に次を決めます。

1. どのようなプレイヤー体験を作りたいか。
2. ラン記録へ何を残すか。
3. 手動プレイで何を確認するか。
4. 採用、調整、削除をどう判断するか。
5. 固定シードまたはデバッグ状態でどう再現するか。

機能を追加した事実ではなく、狙った判断や操作差が確認できたことを完了条件にします。

## 最低限の比較範囲

- 短時間、通常、長時間の手動ラン。
- 開始武器ごとのラン。
- 固定シードの基準版と変更版。
- デバッグと自動テストのランキング除外。
- 15分以上のフレーム、敵数、弾数、ログ容量、保存容量。
- ラン単位の平均 / p95 / 最大実フレーム時間、50ms超過率、Phaser実測FPS。
- 有限終了するまでの脅威ティア、崩壊段階、死因。

## 変更規模に応じた確認

| 変更 | 必須確認 |
| --- | --- |
| 文言、表示だけ | 型確認、対象E2E、画面比較 |
| 純粋なゲームルール | 単体テスト、バランス回帰テスト |
| 保存、履歴、ランキング | 単体テスト、移行テスト、E2E |
| 武器、敵、ウェーブ | 単体テスト、回帰テスト、手動ラン |
| ボス、ステージ、モード | 上記すべてと長時間確認 |

## v0.5の最終判定

個別チケットのテスト成功だけでv0.5完了とは扱いません。`PH-V05-011`で次をまとめて確認します。

- 型確認、単体テスト、ビルド、E2E、画面比較。
- 短時間、通常、長時間の3本以上の手動ラン。
- 15分以上のフレーム、敵数、弾数、ログ容量、保存容量。
- 破損保存、件数上限、再読込、個別初期化。
- デバッグと自動テストのランキング除外。
- 未解決P0がないこと。

詳細は [v0.5チケット詳細](../../project-management/v05-tickets/) を参照してください。

### 2026-07-11の完了結果

- 型確認、18ファイル・121件の単体テスト、本番ビルドに成功した。
- 機能E2E 25件でCanvas、入力、カーソル、保存の一回性、再読込、破損、容量超過、プロフィールを含む個別初期化、設定、音声、縦横モバイルを確認した。
- デスクトップと縦横モバイルの画面比較18件を確定した。通常Playwrightは43件成功、長時間1件スキップだった。
- 900秒、30fps相当の27,000ステップを完走し、敵、弾、ピックアップ、座標が上限内であることを確認した。
- 実時間ブラウザ耐久1件を15.1分実行し、終盤FPS 15超、JSヒープ512MB未満、敵76体以下、全弾220以下、ピックアップ2000以下、Canvas非空白、コンソールエラーなしを確認した。
- v0.5の通常UIで約98秒、178秒、272秒の3ランを行い、ブラウザ履歴、ランキング、開発ログへ同じ手動記録が1件ずつ保存された。
- 3ランはすべてv0.5、ランダムシード、`manual`起点、ランキング対象であり、デバッグと自動テストは混入していない。
- UIと操作は良好という所感を得た。単調という音響指摘へSEバリエーションと32秒BGMを追加し、全16音源のブラウザ読込をE2Eで確認した。
- 響きの機械監査ではBGMが約-17.3 LUFS、True Peak -2.2 dBFSで、クリップや無音を検出しなかった。
- ログ全体は監査時点で約160KB（通常8件・debug 5件・test 5件）だった。
- 実装上の未解決P0はなく、`PH-V05-011`を完了と判定した。

加速試験と実時間耐久は人間の操作感を代替しないため、v0.5では3ランと所感を別途確認しました。今後のバージョンでも、自動検証と人間の評価を分けます。

## v0.6.5の長時間保証

v0.6.5では「15分生存できること」だけを成功条件にしません。障害物経路、EX循環、武器固有成長、高速弾の連続衝突と外周反射に加え、実フレーム時間の分布も保証対象です。

- 24分を上限とする加速入力で、敵接触・敵弾ダメージを除いてもアリーナ崩壊により有限終了する。
- 同時敵数96、プレイヤー弾と敵弾の合計300、ピックアップ2000を上限とする。
- 敵HPと攻撃力は増え続けるが、敵速度、敵数、出現間隔、遠距離弾速度、同時敵弾256は安全上限を超えない。
- 崩壊開始、最高段階、崩壊ダメージ、最終死因を記録する。
- 経路場の直接追跡、迂回、フォールバック、構築回数と、EXサイクル・自動取得を記録する。
- 集束強化、掃射成立・消費、敵役割別撃破、武器別の複数標的命中を記録する。
- 障害物反射、外周反射、外周面別反射、跳弾後命中を記録する。
- 障害物 / 外周別の跳弾後命中・撃破と、外周面別の跳弾後命中を記録する。
- 実フレーム標本1000以上、p95 34ms以下、50ms超過率1%未満、Phaser実測FPS 15超を満たす。

実績: 37ファイル・208件の単体試験、機能E2E 31件、画面比較25件、本番ビルドが成功しました。加速長時間試験は崩壊による有限終了を維持し、固定10シードのPulse / Spread主要指標差は6%未満です。

初回のv0.6.4実時間試験は最大弾408で合計上限300を超え、失敗しました。内訳の理論値と固定試験から射撃体集中時の敵弾密度と特定し、同時敵弾256の安全上限を追加しました。

v0.6.4の再試験は15.1分を完走し、シミュレーション893.76秒、最大敵86、プレイヤー弾21、敵弾254、合計274、最大ピックアップ1004、最大JSヒープ72.1MB、開始60.22fps、終了60.64fpsでした。Canvas非空サンプル1826、保存容量652バイト、コンソールエラー0件も確認しました。v0.6.5では同じ試験へラン単位のフレーム分位を追加しました。

v0.6.5の再試験は15.1分を完走し、シミュレーション891.53秒、実フレーム54188件、平均16.666ms、p95 16ms、最大33.3ms、50ms以上0件でした。推定60.001fps、Phaser実測60.675fps、外部FPSは開始60.010 / 終了60.654です。最大敵86、プレイヤー弾21、敵弾256、合計274、最大ピックアップ1007、最大JSヒープ66.4MiB、Canvas非空サンプル1799、保存容量652バイト、コンソールエラー0件で、全基準を通過しました。

## v0.6.7 Phaser 4 WebGL耐久

Phaser 4.2.1移行後は、headless Chromeが実GPUではなくSwiftShaderを使うことをrenderer情報から確認しました。機能・画像E2Eはheadlessで続け、性能ゲートだけをheaded ChromeのIntel UHD Graphicsへ分離しています。

最終production耐久は15.1分を完走し、シミュレーション892.84秒、実フレーム58583件、平均15.42ms、p95 29ms、最大69.5ms、50ms超過4件でした。推定64.86fps、Phaser実測32.35fps、外部FPSは開始144.14 / 終了31.56です。最大敵86、プレイヤー弾15、敵弾250、合計257、最大ピックアップ980、最大JSヒープ155.8MB、WebGL非空サンプル1791、保存容量652バイト、コンソールエラー0件で、全基準を通過しました。

同じIntel GPUのdev耐久は平均15.84ms、p95 34ms、最大76.4ms、50ms超過9件、終端外部FPS31.69でした。productionはp95と長フレームを改善しましたが、後半の終端FPSは同等です。build有無よりrenderer、GPU、画面読取設定の差が大きいため、比較時は起動条件を必ず記録します。

性能計装は[Phaser TimeStep](https://docs.phaser.io/api-documentation/class/core-timestep)の`rawDelta`と`actualFps`を使います。[Web Vitals](https://github.com/GoogleChrome/web-vitals)はLCP / INP / CLS、[Long Tasks API](https://www.w3.org/TR/longtasks-1/)は50ms以上のメインスレッド占有を対象とするため、現段階では実行時依存を追加しません。
