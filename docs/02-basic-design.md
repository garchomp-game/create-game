# 基本設計

## 1. コンセプト

`Arena Core` は、閉じたアリーナ内でプレイヤーが敵を倒しながら生存時間を伸ばすゲームである。

ゲームの中心は以下のループである。

1. プレイヤーが移動する
2. プレイヤーが照準方向に射撃する
3. 敵が外周から出現する
4. 敵がプレイヤーを追跡する
5. 弾が敵に当たると敵が消える
6. 敵がプレイヤーに触れるとHPが減る
7. 時間経過で敵が増え、難しくなる
8. HPが0になるとゲームオーバーになる

このゲームは、ライブラリ比較のために意図的にシンプルにする。複雑なゲーム性よりも、各エンジンで基本的な2D/3Dゲーム構造をどれだけ自然に書けるかを観察する。

## 2. 画面設計

### 2.1 論理解像度

基準となる論理解像度は以下とする。

```text
960 x 540
```

このサイズを基準に、プレイヤー、敵、障害物、UIの位置を定義する。

2D実装では、そのままピクセル座標として扱う。

3D実装では、上から見た平面をこの論理座標に対応させる。例えば、`x` と `z` を平面座標として使い、カメラを上方から見下ろす。

### 2.2 画面領域

画面は以下の領域に分かれる。

| 領域 | 内容 |
| --- | --- |
| アリーナ | プレイヤー、敵、弾、障害物が存在する主領域 |
| HUD | HP、スコア、経過時間、実装名を表示 |
| ゲームオーバー表示 | ゲームオーバー時に中央へ重ねる |

### 2.3 見た目の基準

外部画像を使わず、図形で表現してよい。

| 対象 | 推奨表現 |
| --- | --- |
| プレイヤー | 青またはシアン系の円/円柱 |
| 照準方向 | 短い線、砲身、または向きが分かる形 |
| 敵 | 赤またはオレンジ系の円/球 |
| 弾 | 黄色または白の小円/小球 |
| 障害物 | グレーの長方形/箱 |
| アリーナ床 | 暗めの無彩色 |
| 外周壁 | 明るめの線または枠 |

デザイン差は許容するが、視認性を優先する。

## 3. ゲーム状態

### 3.1 状態一覧

ゲームは以下の状態を持つ。

| 状態 | 説明 |
| --- | --- |
| `playing` | 通常プレイ中 |
| `gameOver` | HPが0になり、結果表示中 |

任意で以下を追加してもよい。

| 状態 | 説明 |
| --- | --- |
| `boot` | アセットやエンジン初期化中 |
| `paused` | 一時停止 |
| `title` | タイトル表示 |

ただし、共通仕様上はページ読み込み後すぐ `playing` に入ることを推奨する。

### 3.2 状態遷移

```text
start
  -> playing

playing
  -> gameOver  HP <= 0

gameOver
  -> playing   R key
```

任意で `P` キーによる `paused` を実装してもよい。

## 4. エンティティ設計

### 4.1 プレイヤー

プレイヤーはユーザーが操作する中心エンティティである。

責務:

- 入力に応じて移動する
- アリーナ外に出ない
- 障害物にめり込まない
- 照準方向を保持する
- 射撃クールダウンを管理する
- HPを保持する

### 4.2 弾

弾はプレイヤーの射撃によって生成される。

責務:

- 生成時の照準方向へ直線移動する
- 一定時間で消滅する
- 敵に当たると敵へダメージを与え、自身は消滅する
- 障害物または外周に当たると消滅する

### 4.3 敵

敵はプレイヤーを追跡するエンティティである。

責務:

- アリーナ外周から出現する
- プレイヤー方向へ移動する
- 障害物と衝突する
- 弾に当たるとダメージを受ける
- プレイヤーに接触するとダメージを与える
- HPが0になると消滅し、スコアを加算する

### 4.4 障害物

障害物は固定エンティティである。

責務:

- プレイヤーの移動を妨げる
- 敵の移動を妨げる
- 弾を消滅させる

障害物は移動しない。

### 4.5 HUD

HUDはゲーム状態を表示する。

表示項目:

- HP
- スコア
- 経過時間
- 現在の敵数
- 実装名

## 5. システム構成

### 5.1 共通モジュール概念

実装はライブラリにより異なるが、概念的には以下の構成を持つ。

| 概念 | 役割 |
| --- | --- |
| App/Game | エンジン初期化、メインループ管理 |
| Scene/World | ゲーム画面とエンティティ管理 |
| Input | キーボード、マウス状態管理 |
| Entities | プレイヤー、敵、弾、障害物 |
| Collision/Physics | 衝突判定、接触処理 |
| Spawner | 敵出現制御 |
| HUD | 画面表示 |
| GameState | HP、スコア、時間、状態 |

### 5.2 実装ごとの許容差

各ライブラリで自然な構造が異なるため、以下の差は許容する。

- Sceneクラスを使うか、単一ファイルのループで管理するか
- エンティティをクラス化するか、データ配列で管理するか
- 物理エンジンを使うか、手動衝突判定にするか
- UIをCanvas/WebGL内で描くか、HTML DOMで重ねるか
- TypeScriptを使うかJavaScriptを使うか

ただし、ゲームの外部仕様は揃える。

## 6. 入力設計

### 6.1 キーボード

| 入力 | 動作 |
| --- | --- |
| `W` / `ArrowUp` | 上移動 |
| `A` / `ArrowLeft` | 左移動 |
| `S` / `ArrowDown` | 下移動 |
| `D` / `ArrowRight` | 右移動 |
| `Space` | 射撃 |
| `R` | ゲームオーバー時にリスタート |
| `P` | 任意。一時停止 |

### 6.2 マウス

| 入力 | 動作 |
| --- | --- |
| マウス移動 | 照準方向更新 |
| 左クリック | 射撃 |

### 6.3 入力状態

入力はイベント単発ではなく、現在押されている状態として扱う。

移動と射撃は押しっぱなしで継続する。

## 7. ゲームフロー

### 7.1 初期化

初期化時に以下を行う。

1. 描画領域を作成する
2. 入力を登録する
3. アリーナを作成する
4. 障害物を作成する
5. プレイヤーを中央に配置する
6. HUDを作成する
7. ゲーム状態を初期化する
8. メインループを開始する

### 7.2 プレイ中更新

毎フレーム以下を行う。

1. `deltaTime` を取得する
2. 入力から移動方向を計算する
3. プレイヤーを移動する
4. 照準方向を更新する
5. 射撃入力とクールダウンを確認する
6. 弾を更新する
7. 敵スポーンを更新する
8. 敵をプレイヤー方向へ移動する
9. 衝突判定を処理する
10. 消滅対象を削除する
11. 難易度を更新する
12. HUDを更新する
13. HPが0以下ならゲームオーバーへ遷移する

### 7.3 ゲームオーバー

ゲームオーバー時は、エンティティ更新を停止してもよいし、背景だけ停止状態で表示してもよい。

最低限、プレイヤー操作、敵スポーン、射撃は停止する。

`R` が押されたら、現在のエンティティを破棄して初期状態から再開する。

## 8. 2D実装と3D実装の対応

### 8.1 2D実装

2D実装では、座標系は以下とする。

```text
x: 右方向が正
y: 下方向が正
```

プレイヤー、敵、弾は円で表現する。

障害物はAABB、つまり軸に平行な長方形で表現する。

### 8.2 3D実装

3D実装では、見下ろし視点にする。

推奨座標系:

```text
x: 右方向
z: 奥/下方向
y: 高さ方向
```

2D仕様の `x` は3Dの `x` に、2D仕様の `y` は3Dの `z` に対応させる。

カメラは上方からアリーナ全体を見る。見下ろし角度は完全な真上でも、少し斜めでもよい。

## 9. 比較のための設計制約

### 9.1 乱数

可能であれば、固定シードの疑似乱数を使う。

ただし、ライブラリの実装容易性を妨げる場合は、通常の `Math.random()` を使ってよい。その場合は実装メモに記録する。

### 9.2 数値の共通化

プレイヤー速度、敵速度、弾速、スポーン間隔などは `03-detailed-design.md` に合わせる。

### 9.3 アセット

全実装で外部画像アセットなしでも完成するようにする。

### 9.4 過剰な演出の扱い

パーティクルや画面揺れは実装してよいが、ゲーム本体の視認性を損なってはいけない。
